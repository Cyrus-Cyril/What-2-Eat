- 不同查询的 LLM 耗时波动在合理范围
- 10 并发下零失败，RPS 受 LLM Provider 槽位数限制（每槽 3 并发），符合设计预期

## 三、两种模式对比

| 指标 | 标准模式（LLM） | 极速模式 | 提升倍数 |
|------|----------------|----------|----------|
| **请求总数 (60s)** | 63 | 380 | **6.0×** |
| **成功率** | 100.0% | 100.0% | — |
| **中位数 (P50)** | 4800 ms | 28 ms | **171×** |
| **P90 延迟** | 12000 ms | 40 ms | **300×** |
| **P95 延迟** | 13000 ms | 51 ms | **255×** |
| **P99 延迟** | 18000 ms | 79 ms | **228×** |
| **平均延迟** | 6359 ms | 31 ms | **205×** |
| **吞吐量 (RPS)** | 1.23 | 6.42 | **5.2×** |

1. 极速模式 P50 仅 28ms，比标准模式快 171 倍
2. 同样硬件下，极速模式 RPS 是标准模式的 5.2 倍
3. 两种模式 HTTP 成功率均为 100%，无任何 5xx/4xx 错误
4. 标准模式延迟 100% 由 LLM 外部调用主导，非后端代码问题

## 四、并发能力评估

| 模式 | 并发机制 | 10用户表现 | 评价 |
|------|----------|-----------|------|
| **极速模式** | FastAPI asyncio，无限制 | RPS 6.4，P99 79ms | ⭐⭐⭐⭐⭐ |
| **标准模式** | LLM 每槽 3 并发 × Provider 数 | RPS 1.2，成功率 100% | ⭐⭐⭐ |

标准模式受 LLM Provider 槽位数硬限制（`Semaphore(provider_count × 2)`），这是有意为之的"用吞吐换稳定"策略。
