# 1 引言

## 2.3 约定

数据库设计采用以下命名规范：

### 表命名规范

1. 全部使用小写字母
2. 多单词采用下划线分隔
3. 表名尽量体现业务含义

例如：

```text
restaurant
user_query
user_tag_preference
```

------

### 主键命名规范

统一采用：

```text
id
```

------

### 外键命名规范

统一采用：

```text
xxx_id
```

例如：

```text
user_id
restaurant_id
tag_id
```

------

### 索引命名规范

统一采用：

```text
idx_表名_字段名
```

例如：

```text
idx_user_query_user_id
```

------

## 2.4 专门指导

数据库使用过程中应遵循以下要求：

### 数据录入要求

1. 餐厅数据必须来源于合法地图 API
2. 标签数据需保证统一性与规范性
3. 用户反馈需进行合法性校验

------

### 数据更新要求

1. 推荐结果允许定期清理
2. 餐厅缓存需周期更新
3. 用户偏好需要依据反馈动态更新

------

### 数据维护要求

1. 定期备份数据库
2. 定期检查索引性能
3. 避免无约束的大规模删除操作

------

# 3 结构设计

## 3.1 概念结构设计

数据库主要涉及以下实体：

| 实体     | 含义                 |
| -------- | -------------------- |
| 用户     | 系统使用者           |
| 餐厅     | 推荐候选餐厅         |
| 标签     | 餐厅分类标签         |
| 查询记录 | 用户推荐请求         |
| 推荐记录 | 推荐结果             |
| 用户反馈 | 用户对推荐结果的评价 |
| 用户偏好 | 用户长期偏好画像     |

------

系统 E-R 关系如下：

```text
APP_USER
 ├── USER_QUERY
 │        └── RECOMMENDATION
 │                  └── FEEDBACK
 │
 ├── USER_TAG_PREFERENCE
 │
 └── INTERACTION

RESTAURANT
 ├── RESTAURANT_TAG
 ├── INTERACTION
 ├── RECOMMENDATION
 └── FEEDBACK

TAG
 └── RESTAURANT_TAG
```

------

## 3.2 逻辑结构设计

数据库采用关系模型设计。

核心数据表如下：

| 数据表              | 功能         |
| ------------------- | ------------ |
| tag                 | 标签信息     |
| restaurant          | 餐厅基础信息 |
| restaurant_tag      | 餐厅标签关联 |
| app_user            | 用户信息     |
| user_tag_preference | 用户长期偏好 |
| interaction         | 用户行为记录 |
| user_query          | 用户查询记录 |
| recommendation      | 推荐结果     |
| feedback            | 用户反馈     |

------

主要关系如下：

1. 用户与查询记录：一对多
2. 查询记录与推荐结果：一对多
3. 用户与标签偏好：多对多
4. 餐厅与标签：多对多
5. 用户与反馈：一对多

------

## 3.3 物理结构设计

### a．内存结构设计

数据库采用：

```text
InnoDB Buffer Pool
```

进行缓存管理。

热点数据包括：

1. 热门餐厅
2. 用户画像
3. 推荐结果
4. 标签数据

------

### b．外存结构设计

数据库采用：

```text

```

存储引擎。

主要特点：

1. 支持事务
2. 支持外键
3. 支持行级锁
4. 支持崩溃恢复

------

### c．数据访问方式

数据库访问方式如下：

| 方式    | 说明             |

# 4 运用设计

## 4.1 数据字典设计

数据库数据字典主要包括：

1. 数据项定义
2. 表结构定义
3. 主外键关系
4. 索引定义
5. JSON 字段说明

------

核心字段示例如下：

| 字段名                         | 类型         | 说明         |
| ------------------------------ | ------------ | ------------ |
| restaurant.name                | VARCHAR(100) | 餐厅名称     |
| restaurant.rating              | FLOAT        | 餐厅评分     |
| recommendation.final_score     | FLOAT        | 推荐综合评分 |
| feedback.rating                | INT          | 用户评分     |
| user_tag_preference.preference | FLOAT        | 用户偏好值   |

------

数据库中使用 JSON 字段保存推荐解释信息：

```text
matched_tags
reason_hint
explain_json
```

用于支持：

1. 推荐解释生成
2. AI 推荐分析
3. 推荐可解释性增强

------

## 4.2 安全保密设计

数据库安全设计如下：

### 用户访问控制

1. 普通用户仅允许访问自身数据
2. 管理员具有数据库维护权限
3. 后端接口统一控制数据库访问

------

### 数据安全措施

1. 使用参数化 SQL 防止 SQL 注入
2. 使用 ORM 降低非法拼接风险
3. 重要数据定期备份
4. 限制数据库远程访问权限

------

### 隐私保护措施

1. 用户敏感信息最小化存储
2. 不保存用户明文密码
3. 用户行为数据仅用于推荐分析

------

### 后续扩展方向

后续系统可进一步增加：

1. JWT 用户认证
2. 数据库读写分离
3. Redis 缓存隔离
4. 数据加密存储
5. 分布式数据库架构

------
