# 用户手册

## 1 引言

### 1.1 编写目的

本文档用于说明“今天吃什么（What-2-Eat）”软件系统的用途、运行环境、安装初始化方法、输入输出要求、查询方式以及常见出错处理方法，便于项目组成员、课程验收人员和首批试用用户快速完成部署、体验与联调。

预期读者范围如下：

1. 项目开发组成员。
2. 课程指导教师或验收人员。
3. 首批试用用户。
4. 在本地计算机上安装和运行该系统的人员。

### 1.2 背景

a．本用户手册描述的软件系统名称为“今天吃什么（What-2-Eat）智能餐饮推荐系统”。

b．结合当前仓库与现有项目文档，可整理出本项目背景如下：

1. 任务提出者：课程项目教学任务提出方与项目指导教师。
2. 开发者：What-2-Eat 项目开发小组成员。
3. 用户（或首批用户）：项目组成员、课程演示人员、课程验收试用人员。

## 2 用途

### 2.1 功能

结合本软件的开发目的，当前版本实际提供的主要功能如下：

1. 根据用户当前位置获取周边餐馆候选数据。
2. 根据预算、口味、人数、自然语言需求等条件生成推荐结果。
3. 支持自然语言输入的意图识别，并根据场景动态调整推荐逻辑。
4. 返回结构化推荐解释，包括一句话摘要、主要原因、次要原因和分维度说明。
5. 提供健康检查、推荐、反馈、历史查询 4 个后端接口。
6. 提供前端首页、登录注册页、个人偏好页和测试页。
7. 支持反馈记录入库及推荐历史记录查询。

系统功能的极限范围如下：

1. 当前版本主要面向课程项目演示和本地联调，不是生产级商用系统。
2. 当前用户登录注册为前端本地模拟机制，未完成后端真实鉴权。
3. 当前数据库默认使用 MySQL，适合课程项目联调与演示环境部署。
4. 当前推荐结果依赖高德地图接口和网络环境。
5. 大模型能力为可选能力，未配置时系统降级为规则推荐与规则解释。

### 2.2 性能

#### 2.2.1 精度

各项输入输出精度要求如下：

1. 地理坐标输入采用经纬度浮点数，通常保留 6 位小数，满足地图接口调用精度要求。
2. 搜索半径、最大距离、人数等输入为整数。
3. 预算输入为数值型，支持整数或浮点数。
4. 评分输出为浮点数，范围通常在 0 到 1 或 0 到 5 之间。
5. 餐馆距离输出单位为米，按整数返回。
6. 推荐综合得分保留小数，用于排序和解释。

#### 2.2.2 时间特性

根据当前实现和开发环境，系统时间特性如下：

1. `GET /api/health` 为轻量接口，响应时间通常较短。
2. `POST /api/recommend` 的主要耗时来自高德地图数据获取、候选清洗、评分排序和可选 LLM 调用。
3. 当前 LLM 相关请求设置了超时控制，其中意图识别模块使用 3 秒超时保护，避免接口长时间阻塞。
4. 数据库初始化在后端启动时执行一次，启动后不需要重复建表。
5. 历史查询与反馈写入基于 MySQL 数据库，课程项目场景下能够满足联调与演示需求。

#### 2.2.3 灵活性

本软件的适应能力主要体现在以下方面：

1. 支持无意图、模糊意图、明确意图三类输入场景。
2. 支持仅靠参数筛选，也支持自然语言与参数混合输入。
3. 支持通过环境变量切换 API 地址、端口、数据库路径和大模型配置。
4. 支持在未配置 LLM 时自动降级。
5. 支持前端通过 `VITE_API_BASE_URL` 适配不同后端地址。
6. 支持后续扩展 Redis、读写分离和更完整用户系统。

### 2.3 安全保密

当前版本在安全保密方面的设计考虑和实际能力如下：

1. 后端环境变量通过 `.env` 管理，避免将真实 API Key 写死在业务代码中。
2. 当前仓库提供 `.env.example` 作为配置示例。
3. 用户真实注册登录和密码加密机制尚未在后端完整落地，当前前端账号系统仅用于演示。
4. 当前接口默认允许跨域访问，便于开发联调，但不适合直接作为生产环境策略。
5. 系统未实现严格权限分级，因此课程演示环境下应避免接入真实敏感业务数据。

## 3 运行环境

### 3.1 硬设备

运行本软件的最小硬设备配置建议如下：

a．处理机型号与内存：

1. 普通 PC 或笔记本电脑。
2. 建议双核及以上处理器。
3. 建议内存 8GB 及以上。

b．外存储器：

1. 本地磁盘可用空间建议不少于 1GB。
2. 需要能够存放项目源码、Python 依赖、Node.js 依赖和数据库相关脚本。
3. 需要已安装并可访问 MySQL 数据库实例。

c．I/O 设备：

1. 键盘、显示器为必需设备。
2. 浏览器用于访问前端页面和 Swagger 文档。

d．数据传输设备：

1. 需要网络连接以调用高德地图 API。
2. 若启用 LLM 功能，还需保证可访问相应大模型服务接口。

### 3.2 支持软件

运行本软件所需要的支持软件如下：

a．操作系统：

1. Windows、Linux 或 macOS 均可运行。
2. 当前仓库实际开发环境以 Windows 为主。

b．程序语言及运行环境：

1. Python 3.10 及以上版本。
2. Node.js 20 及以上版本。

c．数据库管理系统：

1. 当前版本默认使用 MySQL 8.0 及以上版本。
2. Python 端通过 `SQLAlchemy` 与 MySQL 驱动访问数据库。

d．其他支持软件：

1. `pip` 用于安装后端依赖。
2. `npm` 用于安装前端依赖。
3. 浏览器用于使用前端页面和接口文档。

### 3.3 数据结构

为支持本软件运行，当前需要以下数据库或数据文卷：

| 名称 | 类型 | 作用 |
|------|------|------|
| MySQL 数据库 `what2eat` | 关系型数据库 | 存储查询、推荐、反馈及标签相关数据 |

当前数据库核心表包括：

1. `tag`
2. `restaurant`
3. `restaurant_tag`
4. `user`
5. `user_tag_preference`
6. `interaction`
7. `user_query`
8. `recommendation`
9. `feedback`

## 4 使用过程

软件功能与输入源、输出接收机构之间的关系可概括如下：

| 输入源机构 | 软件功能 | 输出接收机构 |
|------|------|------|
| 用户浏览器页面 | 输入自然语言、预算、口味、人数 | 前端推荐结果页 |
| 前端接口调用 | 向后端提交推荐请求 | 后端推荐结果 JSON |
| 高德地图 API | 提供周边候选餐馆数据 | 后端推荐引擎 |
| MySQL 数据库 | 提供历史记录、偏好记录、反馈存储 | 后端业务模块 |
| 用户反馈输入 | 评分和是否选择餐馆 | 反馈记录与偏好更新 |

### 4.1 安装与初始化

安装与初始化步骤如下。

第一步：进入项目目录。

```bash
cd What-2-Eat
```

第二步：安装后端依赖。

```bash
cd backend
pip install -r requirements.txt
```

第三步：准备 MySQL 数据库。

建议先在 MySQL 中创建业务数据库：

```sql
CREATE DATABASE what2eat DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

然后导入表结构和测试数据：

```bash
mysql -u root -p what2eat < backend/sql/schema_mysql.sql
mysql -u root -p what2eat < backend/sql/seed_mysql.sql
```

第四步：如需启用大模型能力和数据库连接能力，在项目根目录创建 `.env` 文件，并填写必要配置。

```env
LLM_API_KEY=你的API密钥
LLM_API_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
LLM_MODEL=qwen-turbo
AMAP_API_KEY=你的高德地图Key
DB_HOST=127.0.0.1
DB_PORT=3306
DB_NAME=what2eat
DB_USER=root
DB_PASSWORD=你的数据库密码
```

第五步：启动后端服务。

```bash
uvicorn app.main:app --reload
```

系统反应与答复：

1. 后端会连接已配置的 MySQL 数据库。
2. 数据库连接正常时，服务将成功启动。
3. 成功后可访问 `http://localhost:8000/docs` 查看接口文档。

第六步：安装并启动前端。

```bash
cd ../frontend
npm install
npm run dev
```

第七步：在浏览器中访问前端页面，一般为本地 Vite 开发地址。

安装工作完成的测试实例如下：

1. 访问 `http://localhost:8000/docs` 能看到 Swagger 页面，说明后端安装成功。
2. 访问前端首页能够正常打开，说明前端安装成功。
3. 调用 `GET /api/health` 返回状态正常，说明后端服务可用。
4. 在 MySQL 中能查询到业务表，说明数据库初始化成功。

### 4.2 输入

系统输入主要来自浏览器页面表单输入、Swagger 测试输入和数据库中的历史偏好信息。

#### 4.2.1 输入数据的现实背景

a．情况：

1. 用户想查询附近适合当前需求的餐馆。
2. 用户可能改变预算、人数、口味或查询表达方式。

b．情况出现的频度：

1. 推荐请求为随机触发。
2. 偏好调整和注册登录为按需触发。
3. 历史查询为低频查询操作。

c．情况来源：

1. 用户在首页、登录页、偏好页中手工输入。
2. Swagger 文档页中手工输入测试参数。
3. 数据库中已有历史记录和偏好数据。

d．输入媒体：

1. 键盘输入。
2. 浏览器表单控件。
3. HTTP JSON 请求体。

e．限制：

1. 需要保证经纬度、半径、预算和人数等参数格式正确。
2. 当前版本未对所有输入建立严格权限隔离，因此仅建议在开发或演示环境使用。

f．质量管理：

1. 后端使用 Pydantic 对请求参数进行校验。
2. 若输入参数不合法，接口会返回错误信息。
3. 数据清洗模块会过滤无效坐标、缺失字段和异常候选数据。

g．支配：

1. 合法输入进入推荐主流程。
2. 不合法输入由后端拒绝或降级处理。
3. 历史偏好数据由系统内部读取，不要求用户直接输入。

#### 4.2.2 输入格式

当前主要输入为推荐接口请求体，格式要求如下：

a．长度：

1. `query` 为自然语言字符串，长度不宜过长，通常一句到数句即可。
2. `user_id` 为字符串标识。

b．格式基准：

1. JSON 键值对格式。
2. 数值项应符合数字格式要求。

c．标号：

1. 推荐请求中的字段名即输入标识符。

d．顺序：

1. JSON 字段顺序不敏感，但建议按照接口文档顺序填写。

e．标点：

1. 输入遵循标准 JSON 语法。
2. 自然语言 query 可正常使用中文标点。

f．词汇表：

1. 口味词可使用“川菜”“火锅”“烧烤”“轻食”等常见表述。
2. 场景词可使用“聚餐”“约会”“工作日午餐”等表达。

g．省略和重复：

1. `user_id`、`query`、`taste`、`budget_min`、`budget_max` 等字段可按场景省略。
2. 经纬度字段在正常推荐中不建议省略。

h．控制：

1. HTTP 请求开始表示输入开始。
2. 请求体结束即表示输入结束。

#### 4.2.3 输入举例

示例一：推荐接口输入。

```json
{
  "user_id": "u001",
  "query": "想吃点辣的，不要太贵，适合两个人一起吃",
  "longitude": 114.35968,
  "latitude": 30.52878,
  "radius": 1000,
  "max_count": 6,
  "budget_min": 20,
  "budget_max": 60,
  "taste": "川菜",
  "people_count": 2
}
```

说明：

a．控制或首部：HTTP POST 请求，目标接口为 `/api/recommend`。

b．主体：JSON 请求体中的用户需求、位置、预算、人数等字段。

c．尾部：HTTP 请求体结束。

d．省略：`query`、`taste`、`budget_min`、`budget_max` 可在特定场景中省略。

e．重复：同一用户可重复提交多次推荐请求。

### 4.3 输出

系统输出包括接口 JSON 响应、前端推荐展示、错误提示和历史记录结果。

#### 4.3.1 输出数据的现实背景

a．使用：

1. 推荐结果供用户做就餐决策。
2. 历史记录供用户回顾过往推荐。
3. 健康检查结果供开发与运维确认服务状态。

b．使用频度：

1. 推荐结果为高频输出。
2. 历史记录为低频输出。
3. 健康检查为调试型输出。

c．媒体：

1. 浏览器页面显示。
2. Swagger 页面显示。
3. JSON 响应体。

d．质量管理：

1. 后端统一通过响应模型输出字段。
2. 错误情况下返回错误信息，便于定位问题。

e．支配：

1. 推荐结果供前端页面渲染。
2. 历史记录可由用户查看或继续分析。
3. 错误信息主要供用户或开发人员纠正输入与环境配置问题。

#### 4.3.2 输出格式

系统主要输出格式如下：

a．首部：

1. 推荐响应包含 `code` 和 `message`。
2. 健康检查响应包含 `status` 和 `version`。

b．主体：

1. 推荐响应主体为 `recommendations` 列表和可选 `explanation_system`。
2. 历史响应主体为 `data` 列表。

c．尾部：

1. 推荐列表和历史列表的结束即为输出尾部。
2. 历史输出还带有 `total`、`page`、`page_size` 等分页信息。

#### 4.3.3 输出举例

示例一：推荐接口输出。

```json
{
  "code": 0,
  "message": "ok",
  "recommendations": [
    {
      "restaurant_id": "R009",

      "explanation": {
        "summary": "人均约35元，非常合适的中餐厅",
        "reasoning_logic": {
          "primary_factor": "人均价格：人均约35元，非常合适",
          "secondary_factor": "用户口碑：评分4.1，口碑较好"
        },
        "match_details": [
          {
            "dimension": "地理位置",
            "detail": "步行可达",
            "score_impact": "low"
          }
        ],
        "ai_speech": "这家店价格合适，口碑也稳。"
      }
    }
  ]

```

说明：

a．定义：该输出表示当前推荐结果和推荐理由。

b．来源：来源于高德地图候选数据、后端评分逻辑、解释逻辑和可选 LLM 输出。

c．特性：推荐结果为列表结构，字段可直接供前端渲染使用。

### 4.4 文卷查询

本系统当前具有查询能力，主要体现在历史记录查询上。

查询方式如下：

1. 通过 `GET /api/history?user_id=...&page=...&page_size=...` 查询指定用户历史记录。
2. 可在 Swagger 页面直接填写参数并调试。
3. 后续若前端接入历史页，可通过页面完成查询。

初始化与准备要求如下：

1. 后端服务必须已启动。
2. MySQL 数据库中必须存在历史查询与推荐记录。
3. 查询时需要提供合法 `user_id`。

### 4.5 出错处理和恢复

常见出错情况及处理方法如下：

| 出错情况 | 可能原因 | 用户处理方式 |
|------|------|------|
| 后端无法启动 | 依赖未安装、端口冲突、Python 环境异常 | 重新安装依赖，检查端口和 Python 版本 |
| 前端无法请求后端 | 后端未启动或 API 地址错误 | 检查 `VITE_API_BASE_URL` 和后端状态 |
| 推荐结果为空 | 条件过严、附近候选少、地图接口失败 | 放宽预算或口味限制，扩大搜索半径 |
| LLM 解释缺失 | 未配置 LLM 或超时降级 | 检查 `.env` 配置，允许系统使用规则化解释 |
| 历史记录为空 | 数据库中暂无记录 | 先执行推荐请求并生成记录 |

恢复建议如下：

1. 先恢复后端服务，再恢复前端页面访问。
2. 若数据库异常，可重新导入 `schema_mysql.sql` 和 `seed_mysql.sql` 完成恢复。
3. 若环境变量配置错误，修改后重新启动服务。

### 4.6 终端操作

本软件当前以浏览器页面和 Swagger 页面为主要操作终端。

终端配置与连接步骤如下：

1. 启动后端服务后，浏览器访问 `http://localhost:8000/docs` 进行接口级操作。
2. 启动前端服务后，浏览器访问前端首页进行用户级操作。
3. 登录注册、偏好修改、推荐请求和查看推荐结果均通过浏览器完成。

当前终端能力如下：

1. 可通过前端首页进行推荐请求。
2. 可通过前端登录注册页维护本地模拟账号。
3. 可通过前端偏好页修改当前用户偏好。
4. 可通过 Swagger 页面执行健康检查、推荐、反馈和历史查询。

说明：本系统不是传统多终端字符系统，不涉及独立终端设备接入和离线终端控制。
