## 1 引言

### 1.1 编写目的

本文档用于制定"今天吃什么（What-2-Eat）智能餐饮推荐系统"的软件配置管理计划。本计划规定了项目开发过程中对软件配置项进行标识、控制、状态记录与报告、审计与评审的方针和规程，以及配置管理活动所需的组织、工具和方法，以保证软件产品在开发、测试和交付全过程中的完整性、一致性和可追溯性。

本文档预期的读者范围包括：

1. 项目开发组成员 —— 了解并遵守配置管理规程；
2. 项目管理人员 —— 掌握配置项状态和基线进展；
3. 课程指导教师与验收人员 —— 用于评审项目配置管理的规范性；
4. 后续维护人员 —— 了解版本演化历史和配置项清单。

### 1.2 背景

本计划所适用的软件项目的基本信息如下：

- **项目名称**：今天吃什么（What-2-Eat）智能餐饮推荐系统
- **项目类型**：课程项目
- **开发团队**：What-2-Eat 项目开发小组
- **软件功能概述**：基于用户位置与偏好的智能餐饮推荐系统，包含后端 API 服务（FastAPI + Python）、前端 Web 应用（Vue 3 + Vite）和 MySQL 数据库。
- **开发周期**：按照课程教学计划分阶段迭代开发。
- **项目仓库**：使用 Git 进行版本控制，所有源代码、文档和脚本在统一仓库中管理。

## 2 管理

### 2.1 机构

本项目的配置管理活动由项目开发组统一负责，不设独立的配置管理组织。项目组全体成员在日常开发中共同遵循本计划规定的配置管理规程。

项目组中设配置管理员一名（由项目组内指定成员兼任），负责以下协调工作：基线的建立与发布、配置库的维护、配置项变更的审核、配置状态报告的编制。

### 2.2 任务

本计划的配置管理任务覆盖项目开发的全过程，包括：

1. 识别并定义项目中的全部软件配置项；
2. 在各开发阶段建立基线，控制基线的变更；
3. 记录和报告配置项的状态及变更历史；
4. 通过配置审计确保基线的完整性和一致性；
5. 选择和使用适当的工具支撑配置管理活动。

### 2.3 职责

- **配置管理员**：负责维护配置库结构，审批配置项变更请求，建立和发布基线，编制配置状态报告，组织配置审计。
- **开发人员**：遵循本计划规定的配置项标识规范、分支管理策略和变更流程，及时提交代码和文档更新，确保提交信息的完整和准确。
- **测试人员**：基于基线版本开展测试活动，将发现的缺陷记录在对应的配置项上。
- **项目负责人**：监督配置管理活动的执行，在关键阶段节点确认基线的发布。

#### 3.1.2 项目基线

本项目在开发过程中建立以下基线：

- **功能基线**（Function Baseline）：在需求分析阶段结束时建立，包含经评审确认的软件需求规格说明书。本项目的功能基线包含的功能为第 2.2 节所列的 12 项核心功能（F01-F12）。
- **分配基线**（Allocated Baseline）：在概要设计阶段结束时建立，包含经评审确认的推荐系统设计文档、数据库设计文档、解释系统设计文档和接口文档。
- **开发基线**（Development Baseline）：在详细设计阶段结束时建立，包含各模块的详细设计说明和接口定义（具体表现为源代码中的函数签名、Pydantic 模型定义和 API 路由定义）。
- **产品基线**（Product Baseline）：在系统测试完成后建立，包含通过测试的源代码、可运行的构建产物和相关文档。

每条基线在建立时，应对其包含的全部配置项进行版本标记（使用 Git Tag），形成不可变快照。

#### 3.1.3 配置项命名与版本号约定

- 文档类配置项的版本号采用 `vX.Y` 格式，其中 X 为主版本号（重大内容变更），Y 为次版本号（小幅修订或补充）。初始版本为 v1.0。
- 源代码类配置项的版本以 Git 提交哈希值（commit hash）标识，配合 Git Tag 标记基线版本。
- 前端构建产物（`dist/` 目录）的版本号与对应的产品基线版本一致。
- 数据库脚本的版本号与对应基线版本一致。

#### 3.1.4 配置项库

本项目使用 Git 作为配置项库（即版本控制仓库）。仓库结构按以下层次组织：

```
What-2-Eat/               ← 仓库根目录
├── backend/              ← 后端代码（SRC-BE 类）
├── frontend/             ← 前端代码（SRC-FE 类）
├── docs/                 ← 文档（DOC 类）
├── .env.example          ← 环境模板（ENV 类）
└── README.md
```

仓库不纳入版本控制的内容（已配置在 `.gitignore` 中）包括：`.env`（含敏感配置）、`node_modules/`、`__pycache__/`、`dist/`（构建产物）、`*.pyc` 等。

#### 3.1.5 配置项的入库与出库

- **入库**：开发人员通过 `git add` + `git commit` + `git push` 将修改后的配置项提交至远程仓库。提交信息应简明扼要地说明变更内容和原因。
- **出库**：开发人员通过 `git clone` 或 `git pull` 从远程仓库获取当前最新版本的配置项。开展测试或基线发布时，应基于特定的 Git Tag 或分支检出对应版本。

### 3.2 配置控制

#### 3.2.1 变更控制流程

对于已建立基线的配置项，任何变更均需遵循以下流程：

1. **变更申请**：变更发起人提出变更请求，说明变更原因、影响范围和预期的修改内容。
2. **变更评估**：配置管理员（或项目负责人）评估变更的必要性、技术可行性和对已有基线的影响。
3. **变更审批**：经评估可行的变更由项目负责人审批通过。
4. **变更实施**：开发人员基于审批结果实施变更，完成代码修改并通过测试。
5. **变更验证**：配置管理员验证变更后的配置项与变更请求的一致性，确认相关测试通过。
6. **基线更新**：变更纳入对应基线，配置管理员更新基线版本标记（Git Tag），并通知相关成员。

对于尚未建立基线的配置项（开发中的代码和文档），开发人员可自行提交修改，但提交信息应保持清晰完整。

#### 3.2.2 分支管理策略

本项目采用简化的分支管理策略：

- **main 分支**：主分支，存放稳定可运行的最新版本。所有已通过测试的功能最终合并到此分支。
- **功能分支**（feature/xxx）：从 main 分支创建，用于开发新功能或修改现有功能。开发完成后通过 Pull Request 合并回 main 分支。
- **修复分支**（fix/xxx）：从 main 分支创建，用于修复缺陷。修复完成后合并回 main 分支。

#### 3.2.3 配置库权限

- 项目组全体成员对配置库具备读写权限。
- 配置库托管于 Git 平台（本地或远程），main 分支的强制推送（force push）应被禁止或限制为仅项目负责人可执行。
- 远程仓库的访问权限应适当控制，防止未授权的外部访问。

### 3.3 配置状态记录与报告

配置管理员应在关键节点编制配置状态报告，记录以下内容：

1. 各基线的当前版本号和建立日期；
2. 自上一报告周期以来发生的配置项变更清单（变更内容、变更人、变更日期）；
3. 当前处于开发中的功能分支和待合并的变更；
4. 存在的尚未解决的配置项问题（如基线不一致、文档与代码不同步等）。

在课程项目场景下，配置状态报告可以简化为以下形式：

- 在基线建立时使用 Git Tag 进行标记，Tag 信息中记录基线版本和包含的主要变更；
- 在项目阶段性节点（如需求评审、设计评审、代码完成、测试完成）口头或简要书面通报配置状态。

### 3.4 配置审计与评审

配置审计分为两类：

- **功能配置审计**：在基线建立时，审计基线所包含的配置项是否满足需求规格说明书中规定的功能要求。例如，在开发基线建立时，检查所有接口是否均已实现、所有模块是否通过单元测试。
- **物理配置审计**：在基线建立时，审计基线所包含的配置项是否完整，且版本是否正确。例如，检查 Git Tag 所标记的提交是否包含所有预期的文件，是否存在遗漏或多余文件。

在课程项目场景下，配置审计可结合各阶段评审一并进行。例如：

- 需求分析阶段结束时，评审功能基线（需求规格说明书）的完整性和一致性；
- 设计阶段结束时，评审分配基线（设计文档）与需求基线的一致性；
- 编码和单元测试完成后，评审产品基线的完整性和测试通过情况。

## 4 工具、技术和方法

### 4.1 版本控制工具

- 使用 **Git** 作为版本控制系统，所有源代码、文档和配置脚本纳入 Git 管理。
- 远程仓库平台可使用 GitHub、GitLab 或本地 Git 服务器。

### 4.2 文档管理工具

- 需求文档和设计文档采用 Markdown（.md）格式编写，同源代码一起纳入 Git 版本控制。
- Word 文档（.docx）作为部分正式交付文档的格式，也纳入 Git 管理（注意 Git 对二进制文件的 diff 支持有限，需辅助以版本说明）。

### 4.3 构建与依赖管理

- 后端使用 `pip` + `requirements.txt` 管理 Python 依赖，版本号精确到次要版本。
- 前端使用 `npm` + `package.json` / `package-lock.json` 管理 Node.js 依赖，lock 文件确保依赖版本的一致性和可复现性。

### 4.4 配置项检索与跟踪

- 通过 Git 的 `git log`、`git diff`、`git tag` 命令进行版本历史检索和变更追踪。
- GitHub/GitLab 等平台提供的 Issue Tracking 功能可用于配置项问题的跟踪（课程项目中可选）。

## 5 对供货单位的控制

本项目的所有软件配置项均由项目开发组自行开发，不涉及对外部供货单位（如外包开发团队或第三方软件供应商）的配置管理控制。

项目中使用的外部依赖（如 Python 第三方库、Node.js 第三方包、高德地图 API、阿里云 DashScope LLM 服务）属于外部产品或服务，不属于本项目的软件配置项，不纳入本计划的配置管理范围。这些外部依赖的版本通过 `requirements.txt` 和 `package.json` 予以记录和锁定。

## 6 记录的收集、维护和保存

### 6.1 配置管理记录的内容

本项目需收集和维护的配置管理记录包括：

1. 基线建立记录（基线名称、包含的配置项、版本号、建立日期、建立人）；
2. 配置项变更记录（变更的配置项、变更描述、变更人、审批人、变更日期）；
3. 配置审计记录（审计类型、审计日期、审计结果、发现的问题）；
4. 配置状态报告（各周期的配置项状态摘要）。

### 6.2 记录的保存

- 配置管理记录以文档形式存放于项目仓库的 `docs/` 目录中（或独立记录文件中），纳入 Git 版本控制。
- 所有 Git 提交历史本身即构成完整的配置项变更记录，作为配置管理记录的重要组成部分。
- 远程仓库（GitHub/GitLab 或本地 Git 服务器）作为配置管理记录的长期存储介质。
- 在课程项目结束后，建议项目组成员保留一份完整的仓库副本，以防止远程仓库失效导致记录丢失。

### 6.3 记录的维护

- 配置管理员应在每次基线建立和配置审计后及时更新配置管理记录。
- 开发人员应在每次配置项变更的提交信息中清晰说明变更内容，以保证 Git 提交历史作为配置管理记录的有效性。
