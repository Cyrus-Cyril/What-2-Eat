# 📝 “今天吃什么”智能推荐系统·反馈模块设计文档

## 一、 系统目标

反馈系统的核心在于建立**“推荐-反馈-进化”**的闭环机制[cite: 5]。通过收集用户对推荐结果的主动评价，动态调整用户的长期偏好标签（Long-term Preferences），从而解决推荐系统“静态化”的问题[cite: 5]。

> **核心原则：** 极简采集、显式驱动、异步更新、感知进化。

---

## 二、 核心设计思想

### 2.1 显式反馈策略 (Explicit Feedback Only)
为了保证工程可行性并降低数据噪音，本系统舍弃了对“点击率”、“停留时间”等隐式数据的推断，仅依赖用户的**主动表态**[cite: 5]：
*   **点赞 (Like)**：代表用户对该餐厅及其背后标签的高度认可[cite: 5]。
*   **踩 (Dislike)**：代表强烈的负面信号，需触发快速惩罚机制[cite: 5]。

### 2.2 反馈闭环链路
1.  **采集**：用户在前端点击餐厅卡片上的“赞”或“踩”[cite: 4, 5]。
2.  **提取**：后端获取该餐厅关联的所有 `restaurant_tag`[cite: 5]。
3.  **更新**：根据反馈类型，修改 `user_tag_preference` 表中的分值[cite: 5]。
4.  **感知**：下次推荐时，解释系统会在 `hello_voice` 中提及该反馈，建立“我听取了你建议”的信任感[cite: 4, 5]。

---

## 三、 数据库与数据流设计

### 3.1 数据库结构 (交互与偏好)

| 表名 | 关键字段 | 说明 |
| :--- | :--- | :--- |
| **interaction** | `user_id`, `restaurant_id`, `action_type` (LIKE/DISLIKE), `timestamp` | 记录用户的每一次显式表态[cite: 5]。 |
| **user_tag_preference** | `user_id`, `tag_id`, `preference_score` | 存储用户对特定标签的权重分值，由反馈动态修正[cite: 5]。 |

### 3.2 偏好更新算法

当用户产生反馈时，系统对该餐厅关联的每一个标签执行以下更新计算[cite: 5]：

$$Preference(tag) = Preference(tag) + (Action\_Weight \times Tag\_Strength)$$

**权重常量定义：**
*   **点赞 (LIKE)**：`Action_Weight = +0.1`[cite: 5]。
*   **踩 (DISLIKE)**：`Action_Weight = -0.2` (负向信号通常包含更确定的拒绝意图，故权重更高)[cite: 5]。
*   **Tag_Strength**：餐厅在该标签下的原始权重（0~1.0）[cite: 5]。

## 四、 业务逻辑流程



### 4.1 逻辑处理优先级
1.  **即时惩罚**：若用户点击“踩”，该餐厅 ID 应进入该用户的“短期黑名单”，在本次会话或未来 24 小时内不再展示[cite: 5]。
2.  **偏好漂移**：偏好分值的更新应支持**层级传递**。例如，用户赞了一家“川式火锅”，其父标签“川菜”也会获得按衰减比例（如 0.5）计算的加分[cite: 5]。

### 4.2 反馈与解释系统的联动 (AI Speech 触发)
解释系统应在获取推荐列表的同时，检索用户的最近反馈记录，并将其融入话术中[cite: 4]：

*   **正面联动**：
    > “哈咯！**既然你之前给‘川菜’点过赞**，这次我特意帮你找了这几家辣味很正的店，快来看看嘛！”[cite: 4, 5]
*   **负面避雷**：
    > “知道你**不太想吃‘西餐’**，所以我帮你把汉堡披萨都划掉啦，剩下的都是你喜欢的口味咯！”[cite: 4, 5]

---

## 五、 工程实现建议

### 5.1 异步处理 (Background Tasks)
为了不阻塞用户的交互界面，反馈触发的偏好计算应采用**异步任务**[cite: 4, 5]：
1.  前端上报 `action_type`。
2.  后端立即返回 `200 OK` 响应。
3.  启动后台任务（如 FastAPI 的 `BackgroundTasks`）计算标签得分并更新数据库[cite: 5]。

### 5.2 探索与发现 (Exploration)
为了避免系统因反馈过于灵敏而陷入“信息茧房”，在推荐算法计算 `TagScore` 时，应始终保留 **20% 的随机探索权重**，用于推荐用户从未反馈过的新品类[cite: 5]。

## 七、 设计总结

本反馈系统通过**“显式行为采集 + 异步标签修正”**，实现了推荐系统从“静态检索”到“动态进化”的跨越[cite: 5]。它不仅大幅降低了隐式数据追踪的工程复杂度，更通过与解释系统的深度绑定，让用户能够直观地感受到“系统正在变聪明”[cite: 4, 5]。
