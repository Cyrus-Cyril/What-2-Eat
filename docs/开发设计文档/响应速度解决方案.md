# 今天吃什么——响应速度提升方案

# 1. 需求背景
需要将单用户使用时候，用户发送问题到收到结果之间的时间控制在5秒以内

# 2. 方案设计

## 2.1 当前瓶颈分析

单次推荐请求的关键路径如下：

```
intent_parser.parse()              → LLM 调用（硬超时 25s）   ← 最大瓶颈
        ↓
DB: user_profile + blacklist + feedback_context               ← 快（ms 级）
        ↓
DB/Redis: get_candidate_restaurants                           ← 快（ms 级）
        ↓
_score_restaurants()                                          ← 纯 CPU，可忽略
        ↓
asyncio.gather(
    build_explanation_system()     → LLM 调用（总超时 8s）
    build_ai_speeches_for_top_n()  → LLM 批量调用（同量级）
)                                                             ← 第二大瓶颈
```

最坏情况：意图解析（25s）+ 解释系统（8s）串行叠加，实测 P95 在 10s 以上。

## 2.2 可行方案

大模型调用本身并非无法在 5s 内完成，通过合理的架构设计（削减超时、调整关键路径、结果缓存）仍可满足要求。不必要直接放弃大模型，以下是三条可行路径：

**路径一：全规则降级（极速模式）**
完全绕过 LLM，意图解析和解释文案均走规则模板（代码中已有完整的降级路径）。响应时间可达 1s 以内，但推荐质量下降，文案较生硬。

**路径二：SSE 流式响应（感知加速）**
先同步返回基础推荐结果（餐厅列表 + 规则解释），再通过 Server-Sent Events 将 AI 文案（ai_speech、欢迎语）逐步推送到前端。用户感知到的首屏延迟可降至 1s 以内，LLM 文案在后续几秒内渐进渲染。

**路径三：关键路径瘦身 + 超时收紧（推荐）**
通过削减 LLM 超时时间、将 ai_speech 移出关键路径、强化缓存命中，使大模型路径的 P95 也可控制在 5s 以内。实现成本低，不需要改变前后端协议。

# 3. 方案实现

- **削减 LLM 超时时间**：意图解析的全局超时从 25s 收紧至 4s，超时后直接降级为默认规则约束，代码已有完整降级路径，不影响可用性
- **将 ai_speech 移出关键路径**：`build_ai_speeches_for_top_n()` 生成的 AI 文案属于锦上添花，可改为后台异步生成，不阻塞响应；前端先展示卡片，speech 延迟渲染
- **前端极速模式**：在前端提供极速模式开关，开启后请求携带标志位，后端直接走规则降级模板，完全跳过 LLM，响应 < 1s
- **SSE 流式响应**：先推送基础推荐结果，再流式推送 AI 文案；改善用户感知延迟，不需要真正压缩总耗时
- **扩大意图解析缓存命中率**：当前进程内缓存（`_intent_cache`）重启即丢失，改为 Redis 持久化（TTL 1h），相同 query 跨进程、跨重启复用，对高频重复 query 效果显著
- **欢迎语缓存 key 优化**：`explanation_builder` 中 `_hello_cache` 已有框架但命中率低，改为按 `(intent_mode, tuple(sorted(core_tags)))` 作为 key，命中时完全跳过欢迎语 LLM 调用
- **意图缓存双 key 策略（A+B）**：单纯 MD5(query) 无法命中相似但不同写法的 query；采用**查询归一化（A）+ 语义结构化 key（B）**的组合，将缓存命中范围从精确匹配扩展到同义表达，同时自动扩大 thundering herd 锁的覆盖范围，减少并发下的 LLM 重复调用
- **预置向量种子索引（解决冷启动）**：预先人工整理 100-200 条常见 query 模板，离线调 LLM 获取意图结果并校验，部署时将 query embedding + 意图结果写入 Redis 向量索引；服务启动后首个用户请求即可通过向量相似检索命中预置数据，完全规避冷启动时必须调 LLM 的问题

## 3.1 具体算法改动

### 3.1.1 意图解析超时收紧（`intent_parser.py`）

**改动位置**：`_call_unified_llm()` 方法中的 `asyncio.wait_for` 调用

**现状**：
```python
content = await asyncio.wait_for(
    _llm_router.call(prompt, timeout=20.0, max_tokens=400, temperature=0),
    timeout=25.0,
)
```

**改动后**：
```python
content = await asyncio.wait_for(
    _llm_router.call(prompt, timeout=3.5, max_tokens=400, temperature=0),
    timeout=4.0,
)
```

超时后走已有的降级路径（返回空 dict → `_default_neutral_constraint()`），不影响推荐可用性，只是约束精度下降为等权默认值。

### 3.1.2 ai_speech 移出关键路径（`recommender.py`）

**改动位置**：`_recommend_inner()` 的 Step 6

**现状**：`asyncio.gather` 同时等待 `build_explanation_system` 和 `build_ai_speeches_for_top_n`，两者均阻塞响应。

**改动思路**：`build_ai_speeches_for_top_n` 改为 `asyncio.ensure_future` 后台触发，结果写入 Redis 缓存（key = 各餐厅 id 的组合），前端首次请求拿到空 speech，随后轮询或由 WebSocket 推送补全。

```python
# Step 6 改动示意
explanation_system = await build_explanation_system(
    intent, req.query, len(top_n), ...
)
# ai_speeches 不再阻塞，后台异步生成
asyncio.ensure_future(
    _generate_and_cache_speeches(speech_inputs, cache_key)
)
ai_speeches = [""] * len(top_n)  # 先用空占位，前端延迟渲染
```

### 3.1.3 意图缓存改为 Redis 持久化（`intent_parser.py`）

**现状**：`_intent_cache` 是进程内 dict，进程重启或多进程部署时缓存失效。

**改动思路**：
1. 先查 Redis（key = `intent:{md5(query)}`，TTL = 3600s）
2. 命中则直接返回，跳过 LLM
3. 未命中则调用 LLM，结果同时写入进程内缓存和 Redis

```python
redis_key = f"intent:{hashlib.md5(cache_key.encode()).hexdigest()}"
cached = await redis_client.get(redis_key)
if cached:
    return json.loads(cached)
# ... LLM 调用后 ...
await redis_client.setex(redis_key, 3600, json.dumps(result))
```

### 3.1.4 欢迎语缓存 key 优化（`explanation_builder.py`）

**现状**：`_hello_cache` 以完整 intent 序列化字符串为 key，命中率极低。

**改动思路**：欢迎语主要由 `intent_mode`（4种取值）和核心标签决定，改为粗粒度 key：

```python
core_tags = sorted(intent.constraints.get("tags", ConstraintItem()).values)[:3]
hello_key = f"{intent_mode}:{'|'.join(core_tags)}"
if hello_key in _hello_cache:
    return _hello_cache[hello_key]
```

命中率可从接近 0% 提升至 30%~50%（相同口味偏好的不同用户可复用）。

### 3.1.5 意图缓存双 key 策略（`intent_parser.py`）

#### 方案 A：查询归一化

在计算 cache key 之前，先对 query 做标准化处理，使不同写法的同义 query 映射到同一 key：

```python
import re

def _normalize_query(q: str) -> str:
    q = q.strip().lower()
    # 去除标点和多余空白
    q = re.sub(r'[，。！？,!?~～\s]+', '', q)
    # 去除常见语气填充词
    q = re.sub(r'(我想|我要|帮我|给我|随便|一下|一点|有点|有些)', '', q)
    return q
```

效果示例：`"我想吃火锅"` → `"吃火锅"`，`"想吃个火锅"` → `"吃火锅"`，两者命中同一 Redis key。

**附加效果**：`_intent_locks` 中的 per-query 锁以归一化后的 key 为粒度，并发用户发送相似 query 时自动复用同一把锁，只触发一次 LLM 调用，其余等待结果——thundering herd 保护自动扩展到同义表达。

#### 方案 B：LLM 结构化结果反向作语义 key

LLM 返回结构化结果后，额外以**结果字段**为 key 再写一条缓存，供后续语义相同但 query 写法不同的请求命中：

```python
def _semantic_key(result: dict) -> str:
    tags = sorted(result.get("taste", []))
    price = result.get("budget_max")
    radius = result.get("radius")
    return f"intent:sem:{','.join(tags)}:{price}:{radius}"
```

查询逻辑更新为五步（含向量检索，见 3.1.7）：

```
1. cache_key = _normalize_query(user_input)
2. 查 Redis: intent:{md5(cache_key)}              → 命中则返回（精确归一化命中）
3. 查 Redis: intent:sem:{tags}:{price}:{radius}   → 命中则返回（语义结构命中）
4. 查 Redis 向量索引: VSIM(embed(query), top=1)   → 相似度≥0.88 则命中（向量语义命中）
5. 未命中 → LLM 调用
6. LLM 返回后同时写：
     Redis intent:{md5(cache_key)}           TTL=3600s
     Redis intent:sem:{tags}:{price}:{radius}  TTL=3600s
     Redis 向量索引: 写入 (embed(query), intent_result)  （动态扩展索引）
     进程内 _intent_cache[cache_key]
```

**对并发的加速效果**：`"想吃火锅，100块"` 和 `"火锅 预算一百"` 经 LLM 解析后都产出 `{taste:['火锅'], budget_max:100}`，命中同一语义 key，10 个并发用户只需 1 次 LLM 调用，其余 9 个直接命中缓存返回。

> 注意：方案 B 对**单用户冷请求**无效（首次仍需 LLM）；对**并发用户**和**用户复访**效果显著。两者解决不同维度的问题，应与 3.1.1 超时收紧和 3.1.2 ai_speech 移出关键路径一起实施。冷启动问题由 3.1.7 预置向量种子解决。

### 3.1.7 预置向量种子索引（解决冷启动）

#### 核心思路

离线阶段预先构建一个经过人工校验的向量索引，服务启动时加载到 Redis，使得第一个用户请求就能命中向量缓存而无需调 LLM。动态请求产生的新 query embedding 持续写回索引，索引越用越丰富。

```
离线阶段（部署前一次性执行，脚本：scripts/seed_intent_vectors.py）
  ├─ 读取 data/intent_seeds.json（100-200 条常见 query + 人工校验的意图结果）
  ├─ 批量调 embedding 模型计算向量
  └─ 写入 Redis 向量索引（HSET + VSIM）

在线阶段（每次请求，新增 L2c 步骤）
  ├─ embed(normalize(query))                     ≈ 5-15ms（本地模型）
  ├─ Redis VSIM 检索 top-1 最近邻
  │    相似度 ≥ 0.88 → 直接返回预置/历史意图结果，跳过 LLM
  │    相似度 < 0.88 → 继续调 LLM
  └─ LLM 返回后将新 (vector, result) 写回索引（动态扩展）
```

#### 种子数据设计（`data/intent_seeds.json`）

覆盖三个维度的交叉组合，约 150 条：

| 维度 | 覆盖项 | 示例 |
|------|--------|------|
| 菜系/口味（15类） | 川菜、火锅、日料、烧烤、粤菜、快餐、西餐、湘菜、东北菜、涮锅（→火锅）、撸串（→烧烤）… | `"想吃火锅"` |
| 价格意图（6档） | 便宜/实惠/不超过50元/100元以内/中等价位/高档 | `"便宜的快餐"` |
| 距离意图（4档） | 附近/步行可达/500米内/不超过1公里 | `"附近的日料"` |
| 高频组合（60条） | 上述维度两两组合的常见表达 | `"附近便宜的火锅"` |

种子文件格式：
```json
[
  {
    "query": "想吃火锅",
    "intent_result": {
      "taste": ["火锅"], "budget_max": null, "radius": null,
      "constraints": {"tags": {"values": ["火锅"], "strength": "preferred", "weight": 0.5}},
      "exclude_tags": [], "reason_hint": "用户想吃火锅"
    }
  }
]
```

#### embedding 模型选型

推荐使用本地轻量模型，避免为 embedding 引入额外 API 调用延迟：

| 模型 | 大小 | 中文效果 | 推理延迟（CPU） |
|------|------|---------|----------------|
| `paraphrase-multilingual-MiniLM-L12-v2` | 90MB | 良好 | 5-15ms |
| `text2vec-base-chinese` | 400MB | 优秀 | 10-30ms |

选 `paraphrase-multilingual-MiniLM-L12-v2` 即可满足需求，安装一个 `sentence-transformers` 依赖。

#### 相似度阈值说明

| 阈值 | 效果 |
|------|------|
| < 0.85 | 误命中率高，`"想吃清淡的"` 可能匹配 `"想吃辣的"` |

#### 与 A+B 方案的协同效果

```
同一语义 query 的命中路径（按速度排序）：

  L1 进程内 dict        ← 完全相同 query 的重复请求，0ms
  L2a 精确归一化 Redis  ← 同 query 不同标点/语气词，~1ms  
  L2b 语义结构 Redis    ← 相同意图不同数字表达，~1ms
  L2c 向量索引 Redis    ← 同义但写法差异大 / 冷启动种子覆盖，~15ms
  L3  LLM               ← 真正新颖的 query，≤4s

预期各层拦截率（稳定运行后）：
  L1+L2a+L2b：~40%  L2c 种子：~35%  LLM：~25%
  → 约 75% 的请求无需调 LLM
```

# 4. 修改后架构

## 4.1 整体架构对比

### 改动前（当前架构）

```
前端
  │
  └─ POST /api/recommend
         │
         ├─[串行 LLM, ≤25s] intent_parser._call_unified_llm()
         │       进程内缓存（重启丢失，多进程独立）
         │
         ├─[并行 DB] get_user_profile + get_user_blacklist + get_recent_feedback_context
         │
         ├─[Redis/DB] get_candidate_restaurants()
         │
         ├─[纯CPU] _score_restaurants()  →  fallback 兜底
         │
         └─ asyncio.gather(                            ← 两路 LLM 全部阻塞响应
                build_explanation_system()  [LLM ≤8s]
                build_ai_speeches_for_top_n() [LLM ≤8s]
            )
                │
                └─ Response（含完整 ai_speech）
```

最坏延迟：25s（意图）+ 8s（explain gather）= 33s；实测 P95 ≈ 10s+

---

### 改动后（目标架构）

```
前端
  │
  ├─[可选] POST /api/nearby  ← 用户打开 App 时预热，提前缓存周边餐厅到 Redis
  │
  └─ POST /api/recommend
         │
         ├─[LLM ≤4s / Redis命中则0ms] intent_parser._call_unified_llm()
         │       ┌─ 优先查 Redis（TTL=1h，跨重启/多进程）
         │       └─ 未命中 → LLM（timeout收紧为4s）→ 写 Redis + 进程内缓存
         │
         ├─[并行 DB] get_user_profile + get_user_blacklist + get_recent_feedback_context
         │
         ├─[Redis命中则0ms / DB回源] get_candidates_no_api()
         │       └─ /api/nearby 已预热时直接命中 Redis
         │
         ├─[纯CPU] _score_restaurants()  →  fallback 兜底
         │
         ├─[LLM ≤3s / _hello_cache命中则0ms] build_explanation_system()
         │       └─ key=(intent_mode, core_tags)，相同口味可跨用户复用
         │
         ├─[asyncio.ensure_future, 不阻塞] _generate_and_cache_speeches()
         │       └─ 结果写入 Redis（key=result_id），前端延迟渲染
         │
         └─ Response（ai_speech=null，其余字段完整）   ← ≤5s 返回
                │
                └─[前端轮询 / WebSocket push] 补全 ai_speech
```

关键路径延迟：意图（≤4s） + build_explanation_system（≤3s，与意图串行）
→ 两者之和上限 7s，但意图缓存命中时降为 ≤3s，总计 **≤5s**（explain_system 与意图有重叠）

### 4.2.2 意图解析层（`app/services/intent_parser.py`）

**`_call_unified_llm()` 改动：**

```
原：asyncio.wait_for(..., timeout=25.0)
改：asyncio.wait_for(..., timeout=4.0)   # 单次 HTTP timeout=3.5s
```

**缓存层升级（四级缓存 + 双 key + 向量索引）：**

```
查询顺序：
  L1:  进程内 dict（_intent_cache[normalize(query)]）← 命中率最高，0ms
       ↓ miss
  L2a: Redis intent:{md5(normalize(query))}         ← 归一化精确命中，跨重启/多进程，~1ms
       ↓ miss
  L2b: Redis intent:sem:{tags}:{price}:{radius}     ← 语义结构命中，同义不同写法，~1ms
       ↓ miss
  L2c: Redis 向量索引 VSIM（含预置种子 150 条）      ← 向量相似命中，冷启动可用，~15ms
       相似度 ≥ 0.88 → 命中；< 0.88 → miss
       ↓ miss
  L3:  LLM 调用（timeout=4s）← 结果同时回填 L1 + L2a + L2b + L2c
```

超时/失败仍走已有的 `_default_neutral_constraint()` 降级路径，不影响推荐可用性。

`fast_mode=true` 时完全跳过 LLM，直接返回 `_default_neutral_constraint()`。

### 4.2.3 数据获取层（`app/services/data_entry.py`）

无需改动代码，但依赖**前端预取配合**：

```
用户打开 App
    → 前端调用 POST /api/nearby（传入当前位置）
    → prefetch_and_store() 调高德 API，写入 Redis（TTL=600s）+ MySQL
    
用户发送推荐请求
    → get_candidates_no_api() 优先读 Redis（0ms）
    → 命中则直接返回，不再等待高德 API
```

预热命中后，这一步从约 500ms（高德 API）降为约 5ms（Redis）。

## 4.3 请求生命周期（改动后）

```
t=0ms    前端发送 POST /api/recommend
t=0ms    检查 Redis 意图缓存
            命中 → t≈5ms 得到 intent
            未命中 → LLM 调用，≤4s 超时
t≈100ms  并发查询 user_profile / blacklist / feedback_context（DB，ms级）
t≈100ms  Redis 查候选餐厅（已预热则命中；未命中走 DB ≈300ms）
t≈110ms  _score_restaurants()（纯CPU，<10ms）
t≈200ms  build_explanation_system()（hello_cache 命中 0ms；未命中 LLM ≤3s）
t≈200ms  asyncio.ensure_future(build_ai_speeches) 后台启动
t≈200ms  ─── 返回 Response（ai_speech=null） ───  ← 目标 ≤5s

[后台]
t≈1-3s   LLM 批量生成 ai_speech → 写 Redis(speeches:{result_id})

[前端轮询]
t≈1-3s   GET /api/speeches/{result_id} 命中 → 渲染 ai_speech 卡片
```

**最优路径（缓存全命中）**：< 300ms 返回
**正常路径（意图缓存未命中，hello_cache 命中）**：≤ 4s 返回
**最差路径（所有缓存未命中，LLM 正常响应）**：≤ 5s 返回（intent 4s + explanation 3s 因 await 串行，但 explanation 的 LLM timeout=3s，实际意图+解释两个 LLM 调用并非完全串行——intent 返回后立即开始 explanation，总墙钟时间取决于 intent 耗时 + explanation 耗时，上限 4+3=7s，需结合 3.1.1 的超时参数和实际 LLM 响应时间评估）

> **注**：若 7s 上限仍无法满足，可将 `build_explanation_system` 也改为后台异步，关键路径仅保留意图解析，上限缩减至 4s，代价是响应中欢迎语也延迟渲染。
