## 1. 标签系统概述

标签系统是推荐引擎的核心基础设施，承担以下职责：

| 职责 | 说明 |
|------|------|
| **餐厅特征提取** | 将高德 API 返回的非结构化分类信息，转换为系统内部统一的结构化标签 |
| **用户偏好表达** | 用户的口味偏好以标签形式存储，与餐厅标签做匹配计算标签分 |
| **推荐打分依据** | 标签匹配度（TagScore）占综合评分权重的 20%（默认）|
| **偏好持久化** | 用户对餐厅的 LIKE/DISLIKE 通过标签偏好表（`user_tag_preference`）长期积累 |

### 标签类型体系

系统标签按 `type` 字段分为四类，与 `seed_mysql.sql` 中的数据一致：

| type | 含义 | 示例 |
|------|------|------|
| `cuisine` | 菜系 | 中餐、川菜、粤菜、日料、韩餐、西餐 |
| `taste` | 口味 | 辣、清淡、甜、酸、咸鲜 |
| `type` | 餐厅类型 | 火锅、烧烤、快餐、面食、堂食、外卖、自助餐 |
| `scene` | 就餐场景 | 聚餐、约会、商务、家庭、午餐、夜宵 |

标签支持**层级结构**（`tag.parent_id`），例如"川菜"的父标签是"中餐"，用户偏好更新时会同步向父标签传递（衰减比例 0.5）。

## 2. 数据库标签表结构

```sql
-- 标签主表（支持层级）
CREATE TABLE tag (
    id          INT PRIMARY KEY AUTO_INCREMENT,
    name        VARCHAR(100) NOT NULL UNIQUE,  -- 标签名，如"川菜"
    type        VARCHAR(50)  NOT NULL,          -- cuisine/taste/type/scene
    description TEXT,
    parent_id   INT,                            -- 父标签 ID（支持层级，可空）
    FOREIGN KEY (parent_id) REFERENCES tag(id)
);

-- 餐厅-标签关联表（多对多，含权重）
CREATE TABLE restaurant_tag (
    restaurant_id VARCHAR(100) NOT NULL,
    tag_id        INT          NOT NULL,
    weight        FLOAT DEFAULT 1.0,            -- 标签强度（0~1）
    PRIMARY KEY (restaurant_id, tag_id),
    FOREIGN KEY (restaurant_id) REFERENCES restaurant(id),
    FOREIGN KEY (tag_id)        REFERENCES tag(id)
);

-- 用户标签偏好表（用户画像核心）
CREATE TABLE user_tag_preference (
    user_id     VARCHAR(100) NOT NULL,
    tag_id      INT          NOT NULL,
    preference  FLOAT DEFAULT 0.5,             -- 偏好分（0~1）
    updated_at  DATETIME,
    PRIMARY KEY (user_id, tag_id)
);
```

## 3. 高德 API 原始字段分析

高德地图周边搜索（`GET /v5/place/around`）返回的 POI 数据中，与标签相关的字段有两个：

### 3.1 `type` 字段（分类路径）

**位置：** POI 根节点  
**格式：** 分号分隔的层级路径  
**示例：** `"餐饮服务;中餐厅;川菜馆"`

```
餐饮服务          ← 一级大类（通常固定，不含餐厅特征信息）
  └── 中餐厅      ← 二级分类（餐饮大类，如中餐/日料/快餐）
        └── 川菜馆 ← 三级分类（最细分类，最具区分度）
```

**旧版处理问题：** 原代码 `category = type.split(";")[-1]` 只取最后一段（`"川菜馆"`），丢失了二级分类（`"中餐厅"`）信息，导致仅凭 `"中餐厅"` 无法被正确映射为 `"中餐"` 标签。

**新版处理：** 保留完整路径存入 `amap_type_path` 字段，对全路径做关键词匹配，父类和子类均可命中。

### 3.2 `business.tag` 字段（美食特色标签）

**位置：** POI → `business` 子对象  
**格式：** 逗号分隔字符串（仅餐饮类 POI 返回）  
**示例：** `"辣味,家常菜,聚餐,便宜"`

这是高德专为餐饮 POI 标注的特色内容，**直接反映口味和场景信息**，是 `type` 字段无法覆盖的重要补充。

### 两个字段的信息互补关系

| 维度 | `type` 字段能覆盖 | `business.tag` 能覆盖 |
|------|-----------------|---------------------|
| cuisine（菜系） | ✅ 核心来源 | 少量（如"家常菜"） |
| taste（口味）  | ⚠️ 部分（如"川菜"推断"辣"）| ✅ 直接标注（"辣味"） |
| type（类型）   | ✅ 如"快餐"、"火锅" | ✅ 如"堂食"、"外卖" |
| scene（场景）  | ❌ 无法覆盖 | ✅ 直接标注（"聚餐"、"约会"）|

## 4. 两阶段标签提取方案

```
高德 API POI 数据
       │
       ├── type: "餐饮服务;中餐厅;川菜馆"
       │          │
       │          ▼  data_cleaner.clean_restaurant()
       │   amap_type_path = "餐饮服务;中餐厅;川菜馆"（完整保留）
       │   category       = "川菜馆"（最后一段，用于显示）
       │          │
       │          ▼  阶段一  tag_mapper.get_tags(amap_type_path)
       │   关键词匹配 CATEGORY_TAG_MAP
       │   "中餐厅" → ["中餐"]
       │   "川菜馆" → ["川菜","中餐","辣"]
       │   合并去重 → ["中餐","川菜","辣"]
       │
       └── business.tag: "辣味,聚餐,家常菜"
                  │
                  ▼  data_cleaner.clean_restaurant()
           amap_tags = "辣味,聚餐,家常菜"（原始保留）
                  │
                  ▼  阶段二  tag_mapper.get_tags_from_amap_business_tag(amap_tags)
           分段匹配 AMAP_BUSINESS_TAG_MAP
           "辣味" → ["辣"]
           "聚餐" → ["聚餐"]
           "家常菜" → （无匹配，忽略）
           合并去重 → ["辣","聚餐"]
                  │
                  ▼  get_all_tags() 两阶段结果合并
           最终标签 → ["中餐","川菜","辣","聚餐"]
                  │
                  ▼  upsert_restaurants() / save_recommendations()
           restaurant_tag 表写入4条记录（含 weight=1.0）
```

### 核心函数

| 函数 | 文件 | 职责 |
|------|------|------|
| `get_tags(category)` | `tag_mapper.py` | 阶段一：type 路径 → cuisine/type 类标签 |
| `get_tags_from_amap_business_tag(amap_tags)` | `tag_mapper.py` | 阶段二：business.tag → taste/scene 类标签 |
| `get_all_tags(category, amap_tags)` | `tag_mapper.py` | 两阶段合并，对外统一调用入口 |
| `clean_restaurant(raw)` | `data_cleaner.py` | 提取 `amap_type_path` 和 `amap_tags` 字段 |
| `upsert_restaurants(restaurants)` | `crud.py` | 调用 `get_all_tags` 写入 restaurant_tag |
| `save_recommendations(...)` | `crud.py` | 同上，推荐流程中的写入路径 |

## 6. 代码实现流程

### 6.1 数据清洗阶段（`data_cleaner.py`）

```python
def clean_restaurant(raw: dict) -> Restaurant | None:
    # 保留完整 type 路径
    amap_type_path = raw.get("type", "") or ""
    # category 仍取最后一段用于显示
    category = amap_type_path.split(";")[-1].strip() if amap_type_path else "其他"

    business = raw.get("business", {}) or {}
    # 提取 business.tag（仅餐饮 POI 有此字段）
    amap_tags = (business.get("tag") or "").strip()

    return Restaurant(
        ...
        amap_type_path=amap_type_path,  # 新增字段
        amap_tags=amap_tags,            # 新增字段
    )
```

### 6.2 标签提取阶段（`tag_mapper.py`）

```python
def get_all_tags(category: str, amap_tags: str = "") -> list[str]:
    """两阶段合并，对外统一调用入口"""
    tags: set[str] = set()
    tags.update(get_tags(category))                          # 阶段一
    tags.update(get_tags_from_amap_business_tag(amap_tags))  # 阶段二
    return list(tags)

def get_tags(category: str) -> list[str]:
    """阶段一：对 type 路径做关键词匹配 → cuisine/type 类标签"""
    result: set[str] = set()
    cat_lower = category.lower()
    for keyword, tags in CATEGORY_TAG_MAP.items():
        if keyword.lower() in cat_lower:
            result.update(tags)
    return list(result)

def get_tags_from_amap_business_tag(amap_tags: str) -> list[str]:
    """阶段二：对 business.tag 每一段做关键词匹配 → taste/scene 类标签"""
    result: set[str] = set()
    for segment in amap_tags.split(","):
        seg_lower = segment.strip().lower()
        for keyword, tags in AMAP_BUSINESS_TAG_MAP.items():
            if keyword.lower() in seg_lower:
                result.update(tags)
    return list(result)
```

### 6.3 标签写入阶段（`crud.py`）

两个写入路径（预取接口和推荐接口）均使用相同逻辑：

```python
# 两阶段标签提取
restaurant_tag_names = {
    r["restaurant_id"]: get_all_tags(
        r.get("amap_type_path") or r.get("category") or "",
        r.get("amap_tags") or "",
    )
    for r in restaurants
}

# 批量查询已有标签
tag_name_to_id = {tag.name: tag.id for tag in db.query(Tag, names)}

# 自动补录 DB 中没有的标签
for tag_name in (all_tag_names - tag_name_to_id.keys()):
    db.add(Tag(name=tag_name, type=get_tag_type(tag_name)))

# 幂等写入 restaurant_tag（PK 冲突不报错）
for restaurant_id, tag_names in restaurant_tag_names.items():
    for tag_name in tag_names:
        db.merge(RestaurantTag(restaurant_id=..., tag_id=..., weight=1.0))
```

**自动补录机制：** 若 `get_all_tags()` 返回的标签在 `tag` 表中不存在（例如 `business.tag` 带来的新标签），系统会自动 INSERT，类型由 `get_tag_type()` 判断。这保证了标签系统的**可扩展性**，无需手动维护种子数据。

## 7. 标签写入数据库流程

```
prefetch_and_store() 或 save_recommendations()
        │
        ▼
upsert_restaurants(restaurants: list[dict])
        │
        ├── [1] db.merge(Restaurant)  → 幂等写入餐厅基础信息
        │         flush()
        │
        ├── [2] get_all_tags(amap_type_path, amap_tags)
        │         → 两阶段提取，得到 restaurant_tag_names: dict[rid, list[tag_name]]
        │
        ├── [3] SELECT tag WHERE name IN (all_tag_names)
        │         → 查询已有标签 ID
        │
        ├── [4] 若存在未入库标签：
        │         INSERT tag (name, type=get_tag_type(name))
        │         flush()  → 回填自增 ID
        │         重新 SELECT 补全 tag_name_to_id
        │
        └── [5] db.merge(RestaurantTag(restaurant_id, tag_id, weight=1.0))
                  → 幂等写入 restaurant_tag
                  commit()
```

**幂等保证：** `restaurant_tag` 表的主键为 `(restaurant_id, tag_id)`，使用 `MERGE`（相当于 `INSERT OR REPLACE`）操作，同一餐厅重复 prefetch 不会产生重复数据。

## 8. 标签在推荐引擎中的使用

### 8.1 打分阶段（`scorer.py`）

`TagScore` 的计算融合了两个来源：

```
TagScore = min(1.0, matched_score / total_pref)

matched_score = Σ combined_prefs[rtag]   （rtag ∈ restaurant_tags ∩ user_prefs）

combined_prefs:
  当有意图时：当前意图权重 0.7 + 历史偏好权重 0.3
  仅有历史时：历史偏好 100%
  无任何偏好：取中性分 0.5
```

`restaurant_tags` 来自 `tag_mapper.get_tags(category)`（通过 `category` 最后一段匹配）。  
`user_tag_preferences` 来自 `user_tag_preference` 表，通过 `user_profile.get_user_profile()` 加载。

### 8.2 偏好更新阶段（`user_profile.py`）

用户 LIKE/DISLIKE 后，通过 `restaurant_tag` 表获取该餐厅关联的标签，更新 `user_tag_preference`：

```
Preference(tag) = clamp(Preference(tag) + ActionWeight × TagStrength, 0.0, 1.0)

ActionWeight:  LIKE → +0.1  |  DISLIKE → -0.2
TagStrength:   restaurant_tag.weight（通常为 1.0）

层级传递（parent_id 非空时）：
  父标签 Preference += ActionWeight × TagStrength × 0.5（PARENT_DECAY）
```

两阶段标签提取意味着：用户点赞一家"辣味聚餐火锅店"后，`辣`、`聚餐`、`火锅`、`中餐` 等标签的偏好都会同步更新，偏好模型更加准确。

## 9. 标签扩展指南

### 9.1 新增 `CATEGORY_TAG_MAP` 条目

在 `tag_mapper.py` 的 `CATEGORY_TAG_MAP` 中添加：

```python
"新关键词": ["系统标签A", "系统标签B"],
```

要求：
- 关键词需为高德 `type` 路径中实际出现的字符串片段
- 系统标签若为新标签，需同步在 `_TAG_TYPE_MAP` 中声明其 type
- 新系统标签无需手动插入 DB，首次使用时会自动补录

### 9.2 新增 `AMAP_BUSINESS_TAG_MAP` 条目

在 `tag_mapper.py` 的 `AMAP_BUSINESS_TAG_MAP` 中添加：

```python
"高德特色标签关键词": ["系统标签A"],
```

可通过记录线上高德 API 返回的 `business.tag` 字段实际值，持续补充此映射表。

### 9.3 向 `tag` 表增加新标签

对于`seed_mysql.sql` 中未预置的标签，有两种方式：

**方式一（推荐）：** 直接扩展映射表，系统会自动补录 DB。

**方式二（手动）：** 在 `seed_mysql.sql` 中追加 INSERT IGNORE：
```sql
INSERT IGNORE INTO tag (id, name, type, description, parent_id) VALUES
(50, '新标签', 'taste', '说明', NULL);
```

### 9.4 标签层级设置

若新标签是某个现有标签的子类，通过 `parent_id` 建立关联：

```sql
-- "川菜" 是 "中餐" 的子类
INSERT IGNORE INTO tag (name, type, parent_id)
SELECT '川菜', 'cuisine', id FROM tag WHERE name = '中餐';
```

这样用户点赞川菜餐厅后，"中餐"父标签也会自动获得 0.5 倍的偏好加成。
