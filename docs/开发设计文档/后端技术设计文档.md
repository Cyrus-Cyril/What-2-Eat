## 1. 后端总体架构

```
┌─────────────────────────────────────────────────────────┐
│                      FastAPI 应用层                      │
│                  app/main.py  +  app/api/routes.py        │
└────────────────────────────┬────────────────────────────┘
                             │
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
     ┌────────────────┐ ┌──────────────┐ ┌──────────────┐
     │   服务层        │ │  数据模型层   │ │   数据库层   │
     │  services/     │ │  models/     │ │    db/       │
     │  ─ data_entry  │ │  ─ schemas   │ │  ─ database  │
     │  ─ amap_client │ │  ─ restaurant│ │  ─ orm_models│
     │  ─ data_cleaner│ └──────────────┘ │  ─ crud      │
     │  ─ tag_mapper  │                  │  ─ redis_cl  │
     │  ─ intent_parser│                 └──────┬───────┘
     │  ─ scorer      │                         │
     │  ─ recommender │              ┌──────────┴────────┐
     │  ─ explainer   │              ▼                   ▼
     │  ─ explanation_│         ┌─────────┐         ┌────────┐
     │    builder     │         │  MySQL  │         │ Redis  │
     │  ─ user_profile│         └─────────┘         └────────┘
     └────────────────┘
              │
     ┌────────┴──────────┐
     ▼                   ▼
┌──────────┐       ┌──────────┐
│ 高德地图  │       │  通义千问 │
│  AMAP API│       │  LLM API │
└──────────┘       └──────────┘
```

**技术栈：**

| 组件 | 技术选型 |
|------|---------|
| Web 框架 | FastAPI 0.x（异步） |
| 数据库驱动 | SQLAlchemy 2.x 异步（aiomysql） |
| 数据库 | MySQL（通过 mysql_schema.sql 建表） |
| 缓存 | Redis（可选，无 Redis 时降级为无缓存模式） |
| 数据验证 | Pydantic v2 |
| HTTP 客户端 | httpx（异步）/ requests（同步高德调用） |
| LLM | 通义千问（qwen-turbo）兼容 OpenAI 接口 |
| 外部地图 | 高德地图周边搜索 API v5 |

## 4. API 接口文档

**Base URL：** `http://{HOST}:{PORT}/api`  
**交互式文档：** `http://{HOST}:{PORT}/docs`（FastAPI 自动生成 Swagger UI）

所有接口统一返回 JSON。响应体均包含 `code` 字段（`0`=成功，`1`=无结果，`-1`=异常）。

---

### 4.1 `GET /api/health` — 健康检查

**标签：** 系统  
**用途：** 探活，用于负载均衡/监控系统确认服务在线。

**请求参数：** 无

**响应示例：**
```json
{
  "status": "ok",
  "version": "0.2.0"
}
```

**响应字段：**

| 字段 | 类型 | 说明 |
|------|------|------|
| status | string | 固定返回 `"ok"` |
| version | string | API 版本号 |

---

### 4.2 `POST /api/recommend` — 智能推荐

**标签：** 推荐  
**用途：** 核心推荐接口，根据用户位置、偏好、自然语言输入，返回排序后的餐厅推荐列表及解释信息。

**请求体（`RecommendRequest`）：**

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| user_id | string \| null | 否 | null | 用户唯一标识，有值时启用个性化推荐、偏好更新、历史写库 |
| query | string \| null | 否 | null | 自然语言输入，如"想吃辣的便宜的，预算50以内" |
| longitude | float | 否 | 114.362 | 用户当前经度（GCJ-02 坐标系） |
| latitude | float | 否 | 30.532 | 用户当前纬度（GCJ-02 坐标系） |
| radius | int | 否 | 1000 | 搜索半径（米），范围 50~50000 |
| max_count | int | 否 | 10 | 最多返回餐馆数，范围 1~50 |
| budget_min | float \| null | 否 | null | 最低预算（元） |
| budget_max | float \| null | 否 | null | 最高预算（元） |
| taste | string \| null | 否 | null | 口味偏好，如"川菜,火锅" |
| max_distance | int \| null | 否 | null | 最大可接受距离（米），优先于 radius 用于打分 |
| people_count | int \| null | 否 | null | 就餐人数（暂为扩展字段） |

### 4.3 `POST /api/feedback` — 用户反馈

**标签：** 反馈  
**用途：** 接收用户对推荐结果的评价（点赞/踩/星级评分），触发后台偏好模型更新。

**请求体（`FeedbackRequest`）：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| user_id | string | 是 | 用户标识 |
| recommendation_id | string \| null | 否 | 对应推荐记录 ID（用于关联分析） |
| restaurant_id | string | 是 | 餐馆 ID |
| action_type | "LIKE" \| "DISLIKE" \| null | 否 | 显式表态；优先于 rating 推导 |
| rating | int | 是 | 满意度 1~5；≥4 且无 action_type 则视为 LIKE；≤2 则视为 DISLIKE |
| chosen | bool | 否 | 是否实际选择了该餐馆（默认 true） |

**响应体（`FeedbackResponse`）：**

```json
{
  "code": 0,
  "message": "反馈已记录"
}
```

**行为说明：**
1. 立即（同步）写入 `feedback` 表；
2. LIKE/DISLIKE 时额外同步写入 `interaction` 表（用于黑名单查询）；
3. 偏好更新（`update_preference_from_feedback`）以 **BackgroundTask** 方式异步执行，不阻塞响应。

---

### 4.4 `GET /api/history` — 历史记录

**标签：** 历史  
**用途：** 查询指定用户的推荐历史记录，支持分页。

**查询参数：**

| 参数 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| user_id | string | 是 | — | 用户标识 |
| page | int | 否 | 1 | 页码（≥1） |
| page_size | int | 否 | 20 | 每页条数（1~100） |

**响应体（`HistoryResponse`）：**

```json
{
  "code": 0,
  "message": "ok",
  "data": [
    {
      "query_id": "xxx-uuid",
      "restaurant_name": "海底捞火锅",
      "category": "",
      "distance_m": 0,
      "avg_price": 0.0,
      "score": 0.82,
      "created_at": "2026-05-19 12:00:00"
    }
  ],
  "total": 35,
  "page": 1,
  "page_size": 20
}
```

**查询逻辑：** 取该用户所有 `user_query` 记录对应的 `recommendation` 表中 `rank=1`（每次推荐的首位结果）的记录，按创建时间倒序分页返回。

## 5. 服务层模块详解

### 5.1 数据入口服务 `data_entry`

**文件：** `app/services/data_entry.py`

**功能：** 对外统一暴露 `get_candidate_restaurants()` 接口，整合高德 API 调用与数据清洗两个步骤。

**核心函数：**

```python
def get_candidate_restaurants(
    longitude: float, latitude: float,
    radius: int = 1000,
    max_count: int = 20
) -> list[dict]
```

**逻辑流程：**
1. 若 `USE_MOCK=true`，直接返回 `mock_data.py` 中的测试数据；
2. 否则调用 `amap_client.fetch_nearby_restaurants()` 获取高德原始数据；
3. 将原始数据传入 `data_cleaner.clean_restaurants()` 清洗；
4. 将 `Restaurant` 对象列表转换为 `dict` 列表并返回。

---

### 5.2 高德地图客户端 `amap_client`

**文件：** `app/services/amap_client.py`

**功能：** 封装高德地图周边 POI 搜索接口调用，结果缓存至 Redis。

**核心函数：**

```python
def fetch_nearby_restaurants(
    longitude: float, latitude: float,
    radius: int = DEFAULT_RADIUS,
    page_size: int = DEFAULT_PAGE_SIZE
) -> list[dict]
```

**逻辑流程：**
1. 构造 Redis 缓存 Key：`nearby:{lng:.3f}:{lat:.3f}:{radius}:{page_size}`；
2. 若 Redis 命中，直接返回缓存的 POI 列表；
3. 若未命中，构造请求参数（key/location/radius/types=050000/show_fields=business），调用高德 v5 周边搜索接口；
4. 请求超时 10 秒；接口异常或 `status != "1"` 时返回空列表；
5. 成功后将 POI 列表以 **TTL=600秒（10分钟）** 写入 Redis；
6. 返回原始 `pois` 列表。

**Redis Key 示例：** `nearby:114.360:30.529:1000:20`

---

### 5.3 数据清洗器 `data_cleaner`

**文件：** `app/services/data_cleaner.py`

**功能：** 将高德 POI 原始字段映射为系统内部 `Restaurant` 数据类。

**核心函数：**

```python
def clean_restaurant(raw: dict) -> Restaurant | None
def clean_restaurants(raw_list: list[dict]) -> list[Restaurant]
```

**字段映射规则：**

### 5.5 意图解析器 `intent_parser`

**文件：** `app/services/intent_parser.py`

**功能：** 将用户自然语言输入（`query` 字段）解析为结构化的约束集（`IntentConstraint`），通过 LLM 完成，失败时降级为默认约束。

**核心数据结构：**

```python
@dataclass
class ConstraintItem:
    values: list[str]         # 标签值列表（tags 维度）
    preferred: float | None   # 期望值（price，元）
    max_limit: float | None   # 硬性上限（price/distance）
    strength: str             # required / preferred / neutral
    weight: float             # 维度权重 0~1
    tolerance: str | None     # high/medium/low（price 容忍度）

@dataclass  
class IntentConstraint:
    intent_type: str          # "adaptive_constraint"
    constraints: dict[str, ConstraintItem]  # 各维度约束
    exclude_tags: list[str]   # 硬排除标签
    reason_hint: str          # LLM 意图总结
    fallback_applied: bool    # 是否触发了兜底降级
```

**LLM Prompt 设计（Unified Prompt）：**

一次 LLM 调用同时完成三件事：
- **Part A：** 提取基础参数（budget_max/min、radius、taste、scene）
- **Part B：** 提取各维度约束集（strength/weight/values/max_limit/tolerance）
- **Part C：** 提取排除标签和意图总结

**约束强度判断规则：**

| strength | 触发词 | 惩罚曲线 |
|---------|-------|---------|
| `required` | 必须、一定要、只能、不要 | 陡峭 S 型曲线（超 10% 急剧下降）|
| `preferred` | 想吃、喜欢、稍微、最好是 | 平缓指数衰减 |
| `neutral` | 用户未提及 | 无惩罚（系数 1.0）|

**历史上下文融合：** 调用时可传入 `history_context`（最近 3 次推荐摘要），LLM 会参考历史偏好调整解析结果。

**无输入时：** 返回全 `neutral` 等权默认约束（`distance=0.30, price=0.25, rating=0.25, tags=0.20`）。

**核心方法：**

```python
async def analyze_intent(user_input, context, history_context) -> IntentConstraint
def relax_to_preferred(constraint: IntentConstraint) -> IntentConstraint  # 兜底降级
async def parse(user_id, query, raw_params) -> RecommendRequest           # 完整解析并合并
```

### 5.6 评分引擎 `scorer`

**文件：** `app/services/scorer.py`

**功能：** 计算单个餐厅的四维评分，返回 `ScoreDetail`。

**核心函数：**

```python
def calc_all(
    restaurant: dict,
    req: RecommendRequest,
    user_tag_preferences: dict[str, float] | None,
    weights: dict[str, float] | None,
    penalty_factor: float = 1.0,
    filter_budget_max: float | None = None,
) -> ScoreDetail
```

**四维评分公式：**

| 维度 | 公式 | 说明 |
|------|------|------|
| 距离分 D | `max(0, 1 - distance_m / max_dist)` | max_dist 取 max_distance 或 radius |
| 价格分 P | `exp(-|price - mid_budget| / tolerance)` | tolerance = max(10, (budget_max - budget_min)/2)；无预算时取中性分 0.5 |
| 评分分 R | `rating / 5.0` | 无评分时取 0.4 |
| 标签分 T | 融合当下意图 + 历史偏好（加权匹配率） | 无偏好时取中性分 0.5 |

**标签分融合策略（当同时有意图和历史偏好时）：**
- 历史偏好权重：0.3（`alpha=0.7`，偏好分 × 0.3）
- 当下意图权重：0.7（直接加 0.7）
- 最终标签分 = matched_score / total_pref（归一化，最大 1.0）

**最终综合评分：**

$$FinalScore = (w_D \times D + w_P \times P + w_R \times R + w_T \times T) \times penalty\_factor$$

默认权重：$w_D=0.30, w_P=0.25, w_R=0.25, w_T=0.20$（可由意图解析动态覆盖）

**价格软截断（Soft-Clipping）：**  
当餐厅价格超出原始硬筛选预算上限时，在 10% 缓冲区内线性衰减价格分，超出缓冲区的已由松弛搜索剔除。

### 5.7 推荐引擎 `recommender`

**文件：** `app/services/recommender.py`

**功能：** 推荐系统的主流程控制器，完整协调意图解析、候选获取、过滤、打分、解释生成、写库等所有步骤。

**核心函数：** `async def recommend_async(req: RecommendRequest) -> RecommendResponse`

**弹性约束惩罚计算器 `PenaltyCalculator`：**

按约束强度为每个维度计算惩罚系数（0~1，1=不惩罚）：

| 强度 | 曲线类型 | 公式 |
|------|---------|------|
| required | 陡峭 S 型曲线 | `1 / (1 + exp(15 × (vr - 0.1)))` |
| preferred | 平缓指数衰减 | `exp(-λ × max(0, vr))`，λ 由 tolerance 决定（high=1.0/medium=2.0/low=3.0） |
| neutral | 无惩罚 | 恒为 1.0 |

vr（violation_ratio）= 超出比例，如价格超出 10% 则 vr=0.1。

**最终得分公式：**

$$FinalScore = (\sum_i w_i \times Score_i) \times \prod_j Penalty_j$$

### 5.8 解释生成器 `explainer`

**文件：** `app/services/explainer.py`

**功能：** 根据 `ScoreDetail` 评分明细，通过纯规则方式生成结构化 `ExplainData`（维度解释、推荐理由、决策逻辑）。

**核心函数：** `def build_explain(restaurant, score_detail, taste, budget_max) -> ExplainData`

**四维度解释规则：**

| 维度 | 规则 |
|------|------|
| 地理位置 | ≤300m → "步行5分钟内"；≤800m → "步行可达"；其余显示具体距离 |
| 人均价格 | 分 ≥0.8 → "非常合适"；≥0.6 → "适中"；超预算 → "略超预算" |
| 用户口碑 | 评分 ≥4.5 → "口碑极佳"；≥4.0 → "口碑较好"；其余 → "口碑一般" |
| 品类匹配 | 有匹配标签时显示匹配的标签名；无匹配时不追加提示 |

**score_impact 等级判断：** 分 ≥0.7 → `high`；≥0.4 → `medium`；< 0.4 → `low`

### 5.9 解释构建器 `explanation_builder`

**文件：** `app/services/explanation_builder.py`

**功能：** 生成两类 AI 话术：  
1. **全局开场白 `hello_voice`**：总结本次推荐的整体逻辑；  
2. **各餐厅个体推荐话术 `ai_speech`**：口语化介绍单个餐厅的亮点。

**设计原则：** 计算归规则，表达归 AI。LLM 仅负责文字表达，失败时降级为规则模板。

**核心函数：**

```python
async def build_explanation_system(
    intent, req_query, result_count, fallback_note, recent_feedback_context
) -> ExplanationSystem

async def build_ai_speeches_for_top_n(speech_inputs: list[dict]) -> list[str | None]
```

**`hello_voice` 场景模式：**

| 场景模式 | 触发条件 | 规则模板摘要 |
|---------|---------|------------|
| 精准约束推荐 | 存在 required 约束 | "已把完全符合要求的都筛出来了..." |
| 偏好导向推荐 | 存在 preferred 约束 | "按你的口味倾向优先捞出来了..." |
| 综合评分推荐 | 全 neutral | "给你挑了几家离得近、口碑不错的..." |
| 兜底偏好推荐 | fallback_applied=true | "附近暂时没找到完全符合要求的..." |

**近期反馈融合：** `build_feedback_note()` 将用户近期 LIKE/DISLIKE 标签拼成提示注入 LLM Prompt（如"用户最近赞过川菜"），使开场白更个性化。

**LLM 调用参数：** 超时 3 秒，失败后静默降级为规则模板，不抛出异常，保证可用性。

## 8. 核心功能流程

### 8.1 推荐完整流程

```
POST /api/recommend
        │
        ▼
[路由层] routes.py
  ├── 若 query 非空 → intent_parser.parse() 解析并合并参数到 req
  └── 调用 recommend_async(req)
        │
        ▼
[缓存检查] Redis
  ├── 命中 → 直接返回缓存的 RecommendResponse
  └── 未命中 → 继续执行
        │
        ▼
[Step 0] 意图分析
  ├── req.intent 已存在（parse()预置）→ 直接使用
  └── 否则 → intent_parser.analyze_intent(req.query, history_context)
              ├── LLM 解析 → IntentConstraint（含 strength/weight/exclude_tags）
              └── LLM 失败 → 全 neutral 等权默认约束
        │
        ▼
[Step 1] 用户数据加载（仅 user_id 非空时）
  ├── get_user_profile() → 标签偏好字典
  ├── get_user_blacklist() → 近24h DISLIKE 餐厅集合
  └── get_recent_feedback_context() → {liked_tags, disliked_tags}
        │
        ▼
[Step 2] 候选餐厅获取
  └── get_candidate_restaurants(lng, lat, radius, max_count×3)
        ├── USE_MOCK=true → 返回 mock_data
        └── 否则 → 高德 API → 清洗 → list[dict]
        │
        ▼
[Step 3] 候选过滤
  ├── 黑名单过滤：剔除近24h DISLIKE 的餐厅
  └── exclude_tags 硬排除：剔除用户明确说"不要"的品类
        │
        ▼
[Step 4] 打分（含约束惩罚）
  └── _score_restaurants()
        ├── 对每家候选餐厅：
        │   ├── PenaltyCalculator.compute() 计算各维度惩罚
        │   ├── calc_all() 计算四维基础分（含动态权重）
        │   ├── FinalScore = 基础分 × 累积惩罚
        │   └── build_explain() 生成结构化解释
        └── 按 score 降序排序
        │
        ▼
[Step 5] 兜底检查
  ├── Top-N 全部 score < 0.1 → 触发降级
  │   ├── relax_to_preferred()：所有 required → preferred
  │   └── 重新打分并排序
  └── 正常路径继续
        │
        ▼
[Step 6] 并发生成解释（asyncio.gather）
  ├── build_explanation_system() → 全局 hello_voice（LLM/规则）
  └── build_ai_speeches_for_top_n() → 各餐厅 ai_speech（LLM/规则）
        │
        ▼
[Step 7] 注入 ai_speech 到各餐厅 explain 对象
        │
        ▼
[Step 8] 异步写库（ensure_future，不阻塞响应）
  └── _write_to_db() → save_query() + save_recommendations()
        │
        ▼
[Step 9] 映射为前端公开结构 RecommendationItem
        │
        ▼
[缓存写入] Redis TTL=300秒（5分钟）
        │
        ▼
[更新用户推荐历史] Redis key=user_history:{user_id}，TTL=1小时，保留最近3条
        │
        ▼
返回 RecommendResponse
```

### 8.2 反馈处理流程

```
POST /api/feedback
        │
        ▼
[路由层] routes.py
  ├── 推导 action_type（若未显式传入，rating≥4→LIKE，rating≤2→DISLIKE）
        │
        ▼
[同步写库]
  ├── save_feedback()：写 feedback 表
  └── save_interaction()：LIKE/DISLIKE 时写 interaction 表（供黑名单查询）
        │
        ▼
[异步后台任务] BackgroundTask（不阻塞响应）
  └── update_preference_from_feedback()：更新 user_tag_preference 表
        │
        ▼
返回 FeedbackResponse {code: 0, message: "反馈已记录"}
```

### 8.3 用户偏好更新流程

```
update_preference_from_feedback(user_id, restaurant_id, action_type)
        │
        ▼
查询 restaurant_tag 表，获取该餐厅关联的所有标签及权重（Tag_Strength）
        │
        ▼
对每个标签：
  ├── delta = ActionWeight × TagStrength × 1.0
  ├── 若 user_tag_preference 记录存在 → preference = clamp(preference + delta, 0, 1)
  └── 不存在 → 创建记录，初始值 = clamp(0.5 + delta, 0, 1)
        │
        ▼
层级传递（若 tag.parent_id 非空）：
  └── 父标签同步更新，delta' = ActionWeight × TagStrength × 0.5（PARENT_DECAY）
        │
        ▼
commit() 持久化
```

## 9. 缓存策略

系统使用 Redis 在两个层面缓存，Redis 不可用时自动降级为无缓存模式：

| 缓存层 | Key 模式 | TTL | 说明 |
|--------|---------|-----|------|
| 高德 POI 数据 | `nearby:{lng:.3f}:{lat:.3f}:{radius}:{page_size}` | 600s（10分钟） | 相同位置+范围的附近餐厅列表 |
| 推荐结果 | `recommend:{user_id}:{lng:.3f}:{lat:.3f}:{query}` | 300s（5分钟） | 相同用户+位置+query 的完整推荐响应 |
| 用户推荐历史 | `user_history:{user_id}` | 3600s（1小时） | 最近 3 次推荐摘要（供意图解析参考） |

**注意：** 推荐结果缓存 Key 包含 `query` 字段，不同自然语言输入不会复用缓存。

## 10. 评分算法详解

### 10.1 默认权重与动态权重

系统默认权重：

$$w_D=0.30,\quad w_P=0.25,\quad w_R=0.25,\quad w_T=0.20$$

当用户有自然语言输入时，LLM 会根据用户说话重心分配各维度权重（0~1，各维度之和应为 1.0），动态覆盖默认权重。

### 10.2 四维评分详解

**距离分（D）：**

$$D = \max\left(0,\ 1 - \frac{distance\_m}{max\_dist}\right)$$

- `max_dist` 优先取 `req.max_distance`，否则取 `req.radius`；
- 距离为 0 时得满分 1.0，距离等于或超过 `max_dist` 时得 0 分。

**价格分（P）：**

$$P = \exp\left(-\frac{|price - mid\_budget|}{tolerance}\right)$$

$$mid\_budget = \frac{budget\_min + budget\_max}{2}, \quad tolerance = \max\left(10,\ \frac{budget\_max - budget\_min}{2}\right)$$

- 价格恰好等于中间预算时 P=1.0；
- 偏差越大，分数指数衰减越快；
- 无预算信息时取中性分 0.5。

**评分分（R）：**

$$R = \frac{rating}{5.0}$$

- 无评分数据时取保守值 0.4。

**标签分（T）：**

$$T = \min\left(1.0,\ \frac{\sum_{rtag \in match} combined\_prefs[rtag]}{\sum combined\_prefs}\right)$$

融合策略：
- 有意图时，当前意图权重 = 0.7；历史偏好权重 = 0.3；
- 仅有历史偏好时，历史权重 = 1.0；
- 无任何偏好时，取中性分 0.5。

### 10.3 惩罚曲线

**S 型惩罚曲线（required 强度）：**

$$Penalty = \frac{1}{1 + e^{15 \times (vr - 0.1)}}$$

- 违约比 10% 以内几乎无惩罚；
- 超出 10% 后急剧下降至接近 0。

**指数衰减惩罚（preferred 强度）：**

$$Penalty = e^{-\lambda \times \max(0, vr)}$$

- $\lambda$：tolerance=high → 1.0，medium → 2.0，low → 3.0；
- 衰减较平缓，保留了边界外餐厅的参考价值。

## 12. 弹性约束与兜底降级机制

**问题：** 当用户设置强约束（required）时，附近可能没有完全满足条件的餐厅，导致 Top-N 得分全部接近 0。

**方案：三层弹性降级**

| 阶段 | 操作 | 触发条件 |
|------|------|---------|
| 正常推荐 | 按 required/preferred 惩罚正常打分 | 默认 |
| 兜底降级 | `relax_to_preferred()`：required → preferred | Top-N 全部 score < 0.1 |
| 告知用户 | `hello_voice` 中坦诚说明系统调整 | 兜底触发后，在解释话术中提及 |

**意图也支持预防性放宽：**  
`IntentParser.parse()` 调用时，若附近候选极少，可在打分前主动降级（目前通过兜底机制事后处理）。

## 13. 数据库表结构

### 表关系图

```
app_user ──┬── user_query ──── recommendation
           ├── user_tag_preference ── tag ──── restaurant_tag ── restaurant
           ├── interaction ──────────────────────────────────── restaurant
           └── feedback ──────────────────────────────────────── recommendation
                        └──────────────────────────────────────── restaurant
```

### 各表说明

**`app_user`（用户表）**
- `id`（VARCHAR 100，PK）：用户唯一标识

**`tag`（标签表）**
- `id`（BIGINT，PK，自增）
- `name`（VARCHAR 50，唯一）：标签名，如"川菜"
- `type`（VARCHAR 20）：cuisine / taste / type
- `description`（TEXT，可空）
- `parent_id`（BIGINT，FK→tag.id，可空）：父标签，支持层级

**`restaurant`（餐厅表）**
- `id`（VARCHAR 100，PK）：高德 POI ID
- `name`（VARCHAR 100）
- `category`（VARCHAR 100）：高德原始分类
- `address`（VARCHAR 255）
- `latitude/longitude`（FLOAT）
- `rating`（FLOAT，默认 0.0）
- `avg_price`（FLOAT，默认 0.0）
- `updated_at`（DATETIME）

**`restaurant_tag`（餐厅-标签关联表）**
- `restaurant_id`（VARCHAR 100，FK→restaurant.id，PK）
- `tag_id`（BIGINT，FK→tag.id，PK）
- `weight`（FLOAT，默认 1.0）：标签强度

**`user_tag_preference`（用户标签偏好表）**
- `user_id`（VARCHAR 100，FK→app_user.id，PK）
- `tag_id`（BIGINT，FK→tag.id，PK）
- `preference`（FLOAT，默认 0.5）：偏好分 0~1
- `updated_at`（DATETIME）

**`interaction`（用户交互表）**
- `id`（BIGINT，PK，自增）
- `user_id`（VARCHAR 100，FK→app_user.id）
- `restaurant_id`（VARCHAR 100）
- `action_type`（VARCHAR 20）：LIKE / DISLIKE
- `score`（FLOAT，可空）
- `timestamp`（DATETIME）

**`user_query`（用户查询记录表）**
- `id`（VARCHAR 100，PK）：UUID
- `user_id`（VARCHAR 100，FK→app_user.id，可空）
- `longitude/latitude`（FLOAT）
- `radius`（INT）
- `budget_min/budget_max`（FLOAT）
- `taste`（VARCHAR 100）
- `query_text`（TEXT）：自然语言原文
- `created_at`（DATETIME）

**`recommendation`（推荐记录表）**
- `id`（VARCHAR 100，PK）：UUID
- `query_id`（VARCHAR 100，FK→user_query.id）
- `restaurant_id/restaurant_name`（可空）
- `rank_index`（INT）：排名（1 为最高）
- `final_score`（FLOAT）：综合评分
- `score_distance/price/rating/tag`（FLOAT）：各维度分
- `matched_tags`（JSON）：命中标签列表
- `reason_hint`（JSON）：推荐理由列表
- `explain_json`（JSON）：完整解释数据
- `created_at`（DATETIME）

**`feedback`（反馈表）**
- `id`（VARCHAR 100，PK）：UUID
- `user_id`（VARCHAR 100，FK→app_user.id）
- `recommendation_id`（VARCHAR 100，FK→recommendation.id）
- `restaurant_id`（VARCHAR 100）
- `rating`（INT）：1~5
- `chosen`（INT）：0/1，是否实际选择
- `created_at`（DATETIME）
