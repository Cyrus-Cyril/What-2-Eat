# 意图识别设计文档
### 1. 意图识别系统的重构逻辑：三层解析架构

重构后的意图识别不再只是提取标签，而是通过 LLM 产出一个**“动态约束集（Constraint Set）”**。它将用户输入转化为：**维度（Dimension）+ 目标值（Value）+ 强度（Strength）+ 权重（Weight）**。

#### 1.1 输入与解析层
LLM 需要根据用户原始输入，识别出以下细节，并产出结构化 JSON：

* **维度提取**：识别用户关注的维度（品类 `tags`、价格 `price`、距离 `distance`、评分 `rating`）。
* **强度识别 (`strength`)**：
    * `required`：对应原 Scene C。用户使用了“必须、一定要、只能、不要”等绝对词。
    * `preferred`：对应原 Scene B。用户使用了“想吃、喜欢、稍微、最好是”等建议词。
    * `neutral`：对应原 Scene A。用户未提及，系统将依据环境（时间/历史）自动补全。
* **权重分配 (`weight`)**：LLM 根据用户说话的“重音”分配 0~1 之间的值。

#### 1.2 处理不同情况的策略

| 情况 | 识别策略 | 产出示例（JSON 片段） |
| :--- | :--- | :--- |
| **明确且坚决** (原 C) | 识别为 `required`。 | `"price": {"limit": 50, "strength": "required", "weight": 0.4}` |
| **模糊倾向** (原 B) | 识别为 `preferred`。 | `"tags": {"value": ["辣"], "strength": "preferred", "weight": 0.5}` |
| **多目标冲突** | 识别每个维度的强度，LLM 负责初始权重平衡。 | `"dist": 0.4, "price": 0.4, "rat": 0.2` |
| **否定意图** | 识别为特殊的 `exclude` 模式。 | `"exclude_tags": ["日料"], "strength": "required"` |

### 2. 意图识别系统的产出内容 (Output for Recommender)

意图解析模块应向推荐系统提供一个名为 `IntentConstraint` 的对象。

```json
{
  "intent_type": "adaptive_constraint", 
  "constraints": {
    "tags": {
      "values": ["火锅", "川菜"],
      "strength": "required",
      "weight": 0.6 
    },
    "price": {
      "preferred": 80,
      "max_limit": 100, 
      "strength": "preferred",
      "weight": 0.2,
      "tolerance": "high" 
    },
    "distance": {
      "max_limit": 2000,
      "strength": "neutral",
      "weight": 0.2
    }
  },
  "exclude_tags": ["西餐"],
  "reason_hint": "用户非常渴望火锅，但对价格有一定容忍度"
}
```

---

### 3. 推荐系统需要做的相应更改

为了配合“弹性约束模型”，推荐系统需要重构其评分流水线：

#### 3.1 引入惩罚因子函数 (Penalty Function)
不再在 Step 4 进行硬性 `Filter`（除非物理上不可达），而是在计算 $FinalScore$ 时引入 $Penalty$。
* **代码更改**：增加一个 `PenaltyCalculator` 类，根据 `IntentConstraint` 中的 `strength` 选择衰减斜率。
    * `strength == 'required'`：使用陡峭的 S 型曲线（超一点点，分数断崖式下跌）。
    * `strength == 'preferred'`：使用平缓的指数衰减曲线。

#### 3.2 动态权重融合
* **代码更改**：修改 `FinalScore` 公式。不再使用预设的固定表格（Scene A/B/C 表格），而是**直接使用 LLM 传过来的 `weight`**。
* **计算公式更新**：
    $$FinalScore = (\sum w_{intent} \cdot Score_{dimension}) \times \prod Penalty_{dimension}$$

#### 3.3 结果集保底逻辑 (Fallback)
* **代码更改**：在排序后检查 Top 3 的分数。
* 如果分数过低（例如均低于 0.1），说明“硬要求”无法满足。此时推荐系统应自动尝试**“松弛过滤”**（例如将 `required` 降级为 `preferred` 重新算一遍），并在 `my_logic` 中标记“已放宽要求”供解释系统使用。

---

### 4. 解释系统的配合更改

为了体现重构后的“弹性”，解释系统需要读取 `IntentConstraint` 中的 `strength` 和 `reason_hint`。

* **逻辑调整**：
    * 如果某个餐厅是因为 `tolerance`（容忍度）高才入选的，AI 话术应体现转折：“虽然稍微贵了一点，但它可是这附近最正宗的火锅了！”
    * 如果完全匹配了 `required` 约束，AI 应强调精准度：“完全按你的死命令找的，这家绝对没跑！”

### 总结
通过这种方式，你将原有的 Scene B 和 Scene C 逻辑通过**“强度因子”**进行了数学上的统一。意图识别负责**定性和定量**（给权重、定强度），而推荐系统负责**执行权衡**（算分、惩罚）。这种重构让系统具备了处理“非黑即白”之外的灰色地带的能力。
