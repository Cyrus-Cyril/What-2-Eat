# 意图识别中“硬过滤”过于死板的解决方案
### 1. 核心机制：从“加权得分”转向“约束满足”
当用户提出硬性要求时，系统应采取**“先过滤（Filter）、后惩罚（Penalty）、再打分（Score）”**的三段式处理逻辑[cite: 2]。

*   **前置硬过滤 (Filter)：** 无论用户硬筛选的是什么属性（价格、品类或设施），系统首先在数据库查询阶段将其设为 `WHERE` 子句的硬限制，剔除不合格餐厅[cite: 1, 2]。
*   **非线性乘法惩罚 (Penalty)：** 对于那些处于“边界边缘”的餐厅（如预算微超、距离微远），不直接剔除，而是通过乘法因子极大削减其总分[cite: 2]。
    *   例如：如果用户硬性要求“有停车位”，匹配项权重为 $1.0$，不匹配项权重为 $0.05$。即使该店评分再高，也会因为乘积项被压至底部[cite: 2]。
*   **动态权重漂移 (Weight Drift)：** 系统应根据用户“着重表示”的内容，自动将该维度的权重提升至主导地位（通常为 $0.5 \sim 0.6$），而将其余未提及的属性权重均摊[cite: 2, 3]。

### 2. 针对不同硬筛选属性的处理方案

| 硬筛选属性 | 处理策略 (The "Golden Rule") | 解释系统话术引导 (AI Speech) |
| :--- | :--- | :--- |
| **品类 (如: 火锅)** | **绝对过滤**。不匹配品类的餐厅即使价格/评分再好，也只能在“降级搜寻”时出现[cite: 2, 3]。 | “按您的要求，只看了附近的火锅店哈。”[cite: 3] |
| **价格 (如: 70元下)** | **斜坡过滤 (Soft-Clipping)**。允许 $10\% \sim 20\%$ 的溢出缓冲区，但溢出部分得分指数级下降[cite: 2]。 | “帮你看了，70块以内的火锅没几家，多花10块钱这家的味道好得多！”[cite: 3] |
| **距离 (如: 500米内)** | **距离截断**。超过物理极限（如步行15分钟以上）的餐厅直接标记为“不可达”[cite: 2]。 | “现在的店都在您楼下，走两步就到咯。”[cite: 3] |
| **特定设施 (如: 有车位)** | **布尔加权**。将该属性视为 $1/0$ 开关，未命中的餐厅总分强制乘以 $0.1$[cite: 2]。 | “特意帮你找了带车位的店，停车超方便。”[cite: 3] |

### 3. “多路召回”填充算法：解决“筛选冲突”
当用户同时提出两个硬要求（如：便宜 + 火锅）时，容易出现“满足 A 不满足 B”的冲突。算法应按照以下优先级进行**循环探测**[cite: 2]：

1.  **完美匹配路：** 同时满足所有硬筛选条件（Top 1-2）。
2.  **核心偏移路：** 满足“品类”但“价格”微超（作为“品质之选”）。
3.  **约束放宽路：** 价格满足但“品类”从火锅放宽至川菜/冒菜（作为“省钱之选”）。

---

### 4. 总结：灵活适配的系统公式设计
为了适配各种硬筛选场景，建议将最终评分公式重构为：

$$FinalScore = \left( \sum w'_i \cdot Score_i \right) \times \prod Penalty_j$$

*   **$\sum w'_i$：** 这里的权重是动态的。用户提了哪个属性，那个属性的 $w'$ 就变成最大（如 $0.6$）[cite: 2, 3]。
*   **$\prod Penalty_j$：** 这里的惩罚项针对所有的硬筛选需求。只要有一项不符合，就通过连乘效应将其排名压低，保证其无法排在“虽然某项一般但全符合硬要求”的餐厅前面[cite: 2]。

### 5. 解释系统的“降级”告知
在系统执行了上述逻辑后，如果发现第一名是“微超预算”的，解释系统必须向用户“自白”[cite: 3]：
> “哎呀，你要的那个价位（70元）附近真的没有火锅店了咯。**我自作主张帮你看了看80元的**，这几家评价超赞，值得多花那10块钱！”[cite: 3]

这种**“算法做硬度，解释做软度”**的配合，能完美解决硬筛选带来的死板推荐问题[cite: 3]。
