# 🎙️ “今天吃什么”智能推荐系统·解释模块设计文档

------

## 一、系统目标

解释系统的核心目标在于**将枯燥的算法分值转化为用户可感知、可信任的自然语言**。

> 目标：从“冷冰冰的算法排序”进化为“贴心的导览助手”。

**核心诉求：**

- **建立信任**：向用户展示系统确实理解了其意图（意图感知综述）。
- **降低认知负担**：从多维度评分中提取最关键的 1-2 点核心理由。
- **差异化呈现**：针对不同场景（工作日午餐 vs 周末聚餐）给出不同的语言风格。

------

## 二、核心设计思想

### 2.1 设计原则

- **数据驱动解释**：AI 解释的所有“理由”，必须源自后端计算的 `BaseScore` 和 `TagScore` 原始数据以及用户本身原始提问的倾向。
- **结构化输出**：解释结果应包含结构化 JSON 数据（供前端布局）和自然语言文本（供用户阅读）。
- **逻辑分层**：先进行“意图综述（全局）”，后进行“餐厅解释（个体）”。

### 2.2 解释分层架构

1. **全局层（前置综述）**：AI 对用户本次请求的意图分析及策略说明。
2. **局部层（个体解释）**：AI 对每一个推荐餐厅的具体匹配原因说明。

------

## 四、业务逻辑与计算流程

### 4.1 输入数据流

解释系统需要从推荐引擎获取以下数据：

| **输入项**       | **数据来源**                         | **用途（生活化描述）**                                        |
| ---------------- | ------------------------------------ | ------------------------------------------------------------- |
| **用户原始意图** | 用户输入                             | 决定开场白是"赶时间（午餐）"还是"慢享生活（周末）"风格       |
| **匹配得分详情** | 推荐系统生成`calculate_score` 返回值 | 让 AI 判断该"夸"餐厅的哪个点（是离你近，还是口碑好）         |
| **餐厅特征标签** | `restaurant_tag`                     | 作为"老朋友"对暗号的证据（如"你喜欢的那个辣味"）             |
| **场景模式**     | `analyze_intent` 结果                | 决定解释语气是急促（午餐）还是悠闲（周末），匹配用户当下状态 |

这些部分具体来源于推荐系统设计文档中step7提到的部分

| **输入项**         | **对应推荐系统 Step 7 字段**   | **说明**                             |
| ------------------ | ------------------------------ | ------------------------------------ |
| **匹配得分详情**   | `"scores": { ... }`            | 决定了哪个维度是“Primary Factor”     |
| **标签命中情况**   | `"matched_tags": [ ... ]`      | 用于填充“标签匹配”维度的 Detail 文字 |
| **推荐启发点**     | `"reason_hint": [ ... ]`       | 作为 AI 组织语言的优先级参考         |
| **意图场景(新增)** | 从 Step 2 透传的 `intent_type` | 用于生成全局开场白，解释系统逻辑     |

### 4.2 逻辑处理优先级 (Scene Handling)

- **Scene A (环境感知)**：解释重点在于突出“即时性”（如距离近、快餐）。
- **Scene B (模糊加权)**：解释重点在于突出“偏好命中”（如“为您推荐了您喜欢的川菜”）。
- **Scene C (硬过滤)**：解释重点在于突出“精确筛选”（如“已为您排除了所有非火锅店”）。

## 五、AI 解释生成策略 (Prompt Engineering)

为了保证解释的准确性和稳定性，LLM 的 Prompt 应包含以下规则：

### 5.1 解释员角色 (Persona)

> "你是一个有点小幽默、懂吃又懂生活的美食密友。你说话直白坦诚，不讲大道理，更像是在微信上给好朋友发语音推荐餐厅。"

### 5.2 语言风格指南 (Style Guide)

**禁止出现的公式化词汇：**
- 严禁出现"基于您的偏好"、"匹配度"、"加权计算"、"归一化"等机器味词汇。

**必须遵守的语气规则：**
1. **多用语气助词**：多使用"哈、哇、嘛、咯、吧"，增加对话感，让话术读起来像在聊天。
2. **情绪化表达**（让数据"活"起来）：
   - ❌ 坏例子："距离分数为 0.9，推荐此餐厅。"
   - ✅ 好例子："这家就在你楼下，懒人必备！"
3. **坦诚转折**（弱点不回避，但要给理由）：
   - ❌ 坏例子："价格评分略低，但整体评分可以补偿。"
   - ✅ 好例子："虽然稍微有点小贵，但看在它味道这么正的份上，还是值得奖励自己一顿的。"
4. **必须包含证据链**：每个提到的理由必须来自 `score_impact=high` 的维度数据，不得编造。
5. **禁止幻觉**：严禁描述数据中不存在的属性（如：严禁凭空说"环境好"，除非该标签存在）。

## 五-补充、前端展示策略建议

> 本节供前端开发参考，不影响后端 API 结构。

- **`hello_voice` 展示**：以气泡对话框形式出现在推荐列表顶部，模拟"密友发来的语音"效果。
- **`ai_speech` 展示**：作为餐厅详情卡片上的"店小二留言"，独立于评分数据单独呈现。
- **`score_impact` 转化**：将 `high/medium/low` 转化为前端图标（如🔥小火苗 = high，❤️爱心 = medium），**不要把等级文字直接暴露给用户**，保持界面"有温度感"。

## 七、设计总结

本解释系统遵循“计算归规则，表达归 AI”的解耦思想。

通过“意图综述 + 个体解释”的双层架构：

1. **全局综述**让用户感受到被理解（解决了“为什么推荐这些”）。
2. **个体解释**让用户做出判断（解决了“为什么要选这家”）。

这种设计不仅满足了工程实现的可控性，更在用户体验层面赋予了系统“拟人化”的智能感。

------

**一句话总结：**

> 本系统通过结构化的解释输出和前置的意图感知综述，实现了推荐结果从“数据展现”到“智能决策服务”的飞跃。
